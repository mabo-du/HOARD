HOARD: Heritage Observation And Report Drafter
Technical Design Document
Local Multi-Stage Pipeline · 6 GB VRAM Target · v2.0
 
Revised: 9 May 2026  ·  Status: Draft for Review

Document Purpose & Scope
This document is the authoritative technical design for the Excavation Report Drafter: a fully local, multi-stage AI pipeline that converts raw field data — handwritten context sheets, finds catalogues, site photographs, section drawings, and sample results — into a near-publication-ready grey literature report conforming to the relevant national or regional heritage authority standard.
This revision substantially expands the original outline to provide: concrete intermediate data schemas, per-model VRAM budgets, confidence-gated review triggers, fallback strategies, a jurisdiction template engine specification, a CLI interface design, error handling policies, and a testing plan. A developer reading this document should be able to produce a working implementation without further scoping research.
ℹ  NOTE	The pipeline is intentionally modular. Each phase writes its output to disk before the next phase loads. This allows models to be loaded and cleared sequentially, keeping peak VRAM within the 6 GB envelope, and allows individual phases to be re-run without repeating earlier work.


System Architecture Overview
The pipeline consists of five discrete phases plus a final assembly step. Each phase is an independent Python process invoked by a CLI orchestrator. All intermediate outputs are written to a structured working directory (./erd_workspace/{project_id}/).

Phase	Name	Input	Output	Primary Model(s)
0	Ingestion & Triage	Raw files (PDF, JPG, PNG, CSV)	File manifest JSON + quality flags	Rule-based + PaddleOCR-VL
1	Multi-Modal Digitisation	Scanned images, photos	Structured JSON + Markdown tables	TrOCR / Chandra OCR 2 / MinerU
2	Spatial Reconstruction	Plans, sections, drawings	CadQuery Python + annotated images	Zero-To-CAD + Gemma 4-E2B
3	Synthesis & Drafting	All Phase 1-2 outputs	Structured draft (Markdown)	Qwen3.5-4B (Thinking Mode)
4	Compliance Refinement	Draft + jurisdiction template	Compliant report (Markdown/DOCX)	Gemma 4-E2B
5	Assembly & Export	Refined report + assets	Final DOCX + PDF + archive package	Rule-based + pandoc

Working Directory Structure
erd_workspace/
  {project_id}/
    00_manifest/        # Phase 0 file manifest and quality report
    01_digitised/       # Phase 1 JSON outputs per source document
    02_spatial/         # Phase 2 CadQuery scripts and annotated images
    03_draft/           # Phase 3 Markdown draft sections
    04_refined/         # Phase 4 jurisdiction-compliant Markdown
    05_final/           # Final DOCX, PDF, archive ZIP
    assets/             # Copied/normalised input images for embedding
    logs/               # Per-phase logs with model confidence scores
    pipeline_state.json # Resumable pipeline state

Resumability
Each phase checks pipeline_state.json before running. If a phase is marked "complete" it is skipped. This allows the pipeline to be re-started after a crash or power cut without re-running expensive inference steps. Individual phases can be force-re-run with the --rerun-phase N flag.



PHASE
0	Ingestion & Triage
Normalise inputs, assess quality, flag problems before expensive inference begins

Purpose
Before any model is loaded, all input files are inventoried, normalised, and assessed for quality. This phase is entirely rule-based and costs zero VRAM. Its job is to catch problems early — corrupt files, unreadable scans, missing mandatory fields — so that the researcher is informed before the pipeline spends 20 minutes on Phase 1.
Input Specification
The pipeline accepts the following input types, placed in an ./input/ directory or specified individually via CLI flags:
File Type	Expected Source	Format(s) Accepted	Mandatory?
Context sheets	Site recording forms (handwritten or printed)	JPG, PNG, PDF, HEIC	Yes
Finds catalogue	Finds register (typed CSV or handwritten form)	CSV, XLSX, JPG, PNG, PDF	Yes
Site photographs	Field photography	JPG, PNG, HEIC, RAF, NEF	No
Plans	Drawn or CAD-exported site plans	JPG, PNG, PDF, DXF, SVG	No
Section drawings	Drawn section profiles	JPG, PNG, PDF, SVG	No
Sample results	Environmental / dating lab reports	PDF, CSV, TXT	No
Existing text	Previously typed notes or partial draft	TXT, DOCX, MD	No

Triage Steps
    1. Enumerate all files in the input directory and log extensions, sizes, and hashes.
    2. Convert HEIC, RAW, and multi-page PDF inputs to normalised PNG (300 DPI minimum) using Pillow / Wand. Store in 
    3. Run a fast image quality check using OpenCV: detect blur (Laplacian variance < 80 = flag), extreme skew (> 15° = flag), and underexposure (mean pixel value < 40 = flag).
    4. Classify each image as: context_sheet, finds_form, site_photo, plan, section, or unknown using a lightweight classifier (MobileNetV3, < 20 MB, no GPU needed).
    5. For CSV/XLSX finds catalogues: validate column headers against known schema variants; flag missing mandatory columns.
    6. Write the file manifest and quality report to 00_manifest/manifest.json.
    7. Print a human-readable quality summary to stdout and halt if any MANDATORY file is missing or if > 30% of context sheets are flagged as unreadable.

PHASE
1	Multi-Modal Digitisation
Convert physical records into structured JSON and Markdown — the data substrate for all later phases

Overview
This is the most computationally varied phase. Four specialist models handle different document types. They run sequentially, each loading and clearing before the next. The routing decision (which model handles which file) is made deterministically from the manifest type field and the quality flags set in Phase 0.
Model Routing Logic
File Type	Quality Flag?	Routed To	Reason
context_sheet	None	TrOCR + Chandra OCR 2	Clean handwritten form: HTR for text, Chandra for checkbox fields
context_sheet	BLUR / SKEW	PaddleOCR-VL-1.5 first, then Chandra	PaddleOCR corrects distortion before form extraction
finds_form	Any	Chandra OCR 2	Structured form with checkboxes and coded fields
finds_form	CSV	Direct parse (pandas)	No OCR needed for typed CSV — skip model entirely
catalogue_table	Any	MinerU2.5-Pro	Cross-page tables with artifact sketches need table-aware parsing
plan / section	Any	Gemma 4-E2B + Zero-To-CAD	Visual grounding then geometry — handled in Phase 2
site_photo	Any	Gemma 4-E2B	Caption and object identification — handled in Phase 2

1a — Handwritten Text Recognition (TrOCR)
Parameter	Value
Model	microsoft/trocr-base-handwritten (primary)
Fallback model	medieval-data/trocr-medieval-base for pre-1900 scripts
Segmentation	HTRflow page segmenter → line crops → per-line TrOCR inference
Quantisation	FP16 on GPU; INT8 CPU fallback if VRAM unavailable
VRAM footprint	~0.9 GB (FP16)
Confidence output	Per-token log-probabilities → token-level confidence score
Low-confidence threshold	< 0.65 per token → flag for human review
Batch size	16 line crops per inference call

ℹ  NOTE	HTRflow's page segmentation is critical. Do not feed full-page scans directly to TrOCR — it will produce degraded results. HTRflow handles page → region → line segmentation automatically and returns line crops in reading order.

✔  TIP	For modern fieldwork context sheets (post-1980, printed forms with handwritten fills), the standard trocr-base-handwritten model is sufficient. For legacy site archives with older handwriting styles, switch to trocr-base-handwritten-hist-swe-2 or the medieval variant and evaluate on a sample first.

1b — Form & Checkbox Extraction (Chandra OCR 2)
Parameter	Value
Model	datalab-to/chandra-ocr-2
VRAM footprint	~1.2 GB at 4-bit GGUF quantisation
Input	Full-page form image (post-deskew)
Output format	JSON with field labels and values; checkboxes as boolean
Confidence	Per-field confidence in output JSON
Low-confidence threshold	< 0.70 → flag field for human review

1c — Complex Table Parsing (MinerU2.5-Pro)
Parameter	Value
Model	opendatalab/MinerU2.5-Pro
Use case	Multi-page finds catalogues; tables with embedded artifact sketches
VRAM footprint	~1.8 GB
Key capability	Cross-page table merging; in-table image extraction to separate PNG files
Output	Markdown tables + extracted sketch images linked by row ID
When to invoke	Only when input is typed/printed catalogue PDF or photo with visible table grid
Skip condition	If finds catalogue was provided as CSV — parse directly with pandas instead

⚠  RISK	MinerU2.5-Pro should NOT be used for handwritten context sheets — its table-parsing logic will misfire on free-form handwritten text. Route handwritten documents through TrOCR + Chandra instead.

1d — Distortion Correction (PaddleOCR-VL-1.5)
PaddleOCR-VL-1.5 is not used as a primary digitisation model. It is invoked as a pre-processor when the Phase 0 triage has flagged a document with BLUR_LOW, SKEW_HIGH, or EXPOSURE_LOW. Its corrected output is then passed to the appropriate digitisation model (TrOCR or Chandra).
Parameter	Value
Model	PaddlePaddle/PaddleOCR-VL-1.5
VRAM footprint	~0.7 GB (used transiently, then cleared)
Function	Deskew, dewarp, histogram equalise; output corrected PNG
When to invoke	Only when manifest contains BLUR_LOW, SKEW_HIGH, or EXPOSURE_LOW flags
Output	Corrected PNG saved to assets/ alongside original; manifest updated

Phase 1 VRAM Budget
Models load and clear sequentially. Peak VRAM at any one moment is the single largest model.
Step	Model	VRAM (loaded)	Cleared after step?
1d (conditional)	PaddleOCR-VL-1.5	~0.7 GB	Yes — before next step
1a	TrOCR (FP16)	~0.9 GB	Yes — after all HTR complete
1b	Chandra OCR 2 (4-bit GGUF)	~1.2 GB	Yes — after all forms complete
1c (conditional)	MinerU2.5-Pro	~1.8 GB	Yes — after all tables complete
Peak (worst case)	MinerU2.5-Pro alone	~1.8 GB	Well within 6 GB



PHASE
2	Spatial & Technical Reconstruction
Translate site drawings and photographs into technical geometry and annotated visual evidence

Overview
Phase 2 handles the visual interpretation of plans, section drawings, and site photographs. It has two distinct sub-tasks: generating parametric geometry from drawn records (for the archive statement and site model), and producing structured captions and object annotations from site photography (for the narrative and illustration section of the report).
⚠  RISK	Zero-To-CAD geometry generation from field sketches is the most experimental component of this pipeline. Field section drawings are often schematic, inconsistent in scale, and produced under poor conditions. Treat all CadQuery output as a first-approximation to be reviewed and corrected by the site director before inclusion in the final archive. Never include unreviewed geometry in a published report.

2a — Visual Grounding & Photo Captioning (Gemma 4-E2B)
Parameter	Value
Model	unsloth/gemma-4-E2B-it-GGUF (Q4_K_M quantisation)
VRAM footprint	~2.1 GB
Function 1 — Photo captioning	Generate structured caption: what is visible, spatial relationships, scale reference
Function 2 — Visual grounding	For section drawings: identify and label stratigraphic features; output bounding boxes + labels
Function 3 — Cross-check	Compare photo caption to context sheet interpretation; flag inconsistencies
Output	Per-image JSON with caption, grounding labels, and cross-check results
Prompt style	System prompt specifies archaeological domain vocabulary and output JSON schema

Photo Caption Output Schema
// 02_spatial/photo_DSC0042.json
{
  "source_file": "DSC0042.jpg",
  "model": "gemma-4-E2B",
  "caption": "North-facing section through contexts [312] and [374]. Scale rod visible (1m). Clear horizon break at c.0.4m depth. Darker fill [312] overlies paler natural [374].",
  "grounding": [
    { "label": "context_312", "bbox": [0.1, 0.05, 0.9, 0.42], "confidence": 0.91 },
    { "label": "context_374", "bbox": [0.1, 0.42, 0.9, 0.95], "confidence": 0.88 },
    { "label": "scale_rod",   "bbox": [0.82, 0.0, 0.88, 1.0],  "confidence": 0.97 }
  ],
  "cross_check": {
    "matching_contexts": ["[312]", "[374]"],
    "inconsistencies": []
  }
}

2b — Sketch-to-CAD Generation (Zero-To-CAD)
Parameter	Value
Model	ADSKAILab/Zero-To-CAD-Qwen3-VL-2B
VRAM footprint	~1.6 GB
Input	Gemma-grounded section drawing with labelled bounding boxes
Output	CadQuery Python script + rendered SVG preview
Validation	CadQuery script executed in sandbox; if execution fails → flag as GEOMETRY_FAIL
Archive use	Geometry included in archive statement as supplementary; not embedded in main report body
Human review gate	ALL geometry output is flagged REQUIRES_REVIEW before report assembly

✦  DECISION	The Zero-To-CAD output is an archive asset, not a report illustration. The main report body uses the original photographed drawing (or a scanned plan) as the figure. CadQuery geometry is packaged separately for researchers who want a manipulable 3D record.

Phase 2 VRAM Budget
Step	Model	VRAM (loaded)	Cleared after step?
2a	Gemma 4-E2B (Q4_K_M)	~2.1 GB	Yes — after all photos/drawings processed
2b (conditional)	Zero-To-CAD	~1.6 GB	Yes — after all drawings processed
Peak (worst case)	Gemma 4-E2B alone	~2.1 GB	Well within 6 GB



PHASE
3	Synthesis & Narrative Drafting
Reason through the full site dataset and produce a structured first draft

Overview
This is the most cognitively demanding phase and uses the largest model. Qwen3.5-4B in Thinking Mode is given the complete digitised record — all context sheet JSON, finds data, sample results, photo captions, and stratigraphic relationships — and is tasked with producing a structured Markdown draft matching the report skeleton defined by the jurisdiction template (see Section 8).
Model Configuration
Parameter	Value
Model	Qwen/Qwen3.5-4B (Unsloth Dynamic 2.0 GGUF, Q4_K_M quantisation)
VRAM footprint	~2.8 GB at Q4_K_M
Context length	262,144 tokens native; extensible to 1M+ via YaRN RoPE scaling
Thinking Mode	Enabled — model reasons through period sequence before drafting
Inference engine	llama.cpp (server mode) — preferred over vLLM/SGLang for single-GPU local deployment
Context window strategy	Full site data fits comfortably for most excavations; see notes below for large sites
Temperature	0.3 for drafting (low creativity, high fidelity to input data)
Output format	Structured Markdown with section headers matching jurisdiction template

ℹ  NOTE	vLLM and SGLang are excellent for high-throughput server deployments but carry significant overhead (Ray cluster, CUDA graph capture) that is unnecessary and harmful on a single 6 GB consumer GPU. Use llama.cpp in server mode instead — it is specifically optimised for local single-GPU inference and handles the KV cache correctly at this memory level.

Context Assembly Strategy
Before calling the model, the orchestrator assembles a single context document from all Phase 1-2 outputs. This is passed as the user message alongside a system prompt containing the jurisdiction template schema.
# Context document assembly order (Python pseudocode)
context = []
context.append(load_template_schema(jurisdiction))   # ~500 tokens
context.append(site_metadata_block())                # project, grid ref, HER ref, etc.
context.append(render_context_sheet_summary())       # all ctx JSON → condensed table
context.append(render_harris_relationships())        # stratigraphic matrix summary
context.append(render_finds_summary())               # finds catalogue by period/type
context.append(render_sample_results())              # dating, environmental, specialist
context.append(render_photo_captions())              # photo JSON summaries
context.append(render_previous_text())               # any pre-existing typed notes
# Typical total: 15,000–80,000 tokens; 262k limit handles all but the largest multi-season sites

Thinking Mode — What It Does
When Thinking Mode is active, Qwen3.5-4B produces a hidden reasoning chain before generating the output draft. The orchestrator should capture and log this chain to logs/phase3_reasoning.txt — it is extremely useful for the site director to understand how the model interpreted ambiguous stratigraphic relationships or conflicting dating evidence. The reasoning chain is not included in the draft itself.
Draft Structure (Output)
The model is instructed to produce each report section as a separate Markdown code block labelled with the section ID from the jurisdiction template. This allows the orchestrator to parse individual sections for independent compliance checking in Phase 4.
```section:executive_summary
## Executive Summary
An archaeological evaluation was undertaken at...
```

```section:methodology
## Methodology
...
```

```section:results_prehistoric
## Results: Prehistoric Activity
...
```

Human Review Triggers — Phase 3
The orchestrator flags sections for human review if any of the following conditions are met:
    • The model's reasoning chain contains phrases indicating uncertainty (e.g. 'insufficient data', 'conflicting evidence', 'cannot determine').
    • A context sheet referenced in the draft was flagged REVIEW_REQUIRED in Phase 1.
    • The draft references a context number that does not appear in the digitised record.
    • Fewer than 3 dated finds or samples are available to support a stated period attribution.
    • The draft proposes a phasing sequence that contradicts the Harris Matrix stratigraphic relationships.

Phase 3 VRAM Budget
Step	Model	VRAM	Notes
3	Qwen3.5-4B Q4_K_M	~2.8 GB	Largest single load in the pipeline
KV cache (80k token context)	+~0.9 GB	Total ~3.7 GB — within 6 GB
KV cache (262k token context)	+~2.1 GB	Total ~4.9 GB — within 6 GB, leaves ~1.1 GB headroom

⚠  RISK	For very large multi-season sites with > 500 context sheets, the assembled context may approach or exceed the GPU's available KV cache space. In this case: (1) use chunk-and-merge drafting (draft by period, then merge), or (2) offload KV cache layers to CPU RAM using llama.cpp's --n-gpu-layers flag to balance load.



PHASE
4	Compliance Refinement
Ensure the draft conforms to the specific structure, terminology, and conventions of the target jurisdiction

Overview
Phase 4 takes the structured draft from Phase 3 and runs it through a compliance pass driven by a detailed jurisdiction template. Gemma 4-E2B operates in a constrained editing mode: it is explicitly instructed NOT to add interpretive content, only to restructure, relabel, and fill in missing mandatory fields identified by the template schema.
Jurisdiction Template Engine
The compliance engine is driven by a YAML template file for each jurisdiction. New jurisdictions can be added without touching the pipeline code. Templates are stored in ./templates/{jurisdiction_code}.yaml.

Template Schema
# templates/historic_england_cl3.yaml
jurisdiction: "Historic England — Evaluation (CL3 compliant)"
version: "2024"
mandatory_sections:
  - id: executive_summary
    label: "Executive Summary"
    max_words: 300
    required_fields: [project_name, ngr, dates, methodology_summary, key_findings]
  - id: methodology
    label: "Methodology"
    required_fields: [trench_strategy, recording_system, finds_policy, sample_policy]
  - id: results_by_period
    label: "Results"
    sub_sections: [prehistoric, roman, medieval, post_medieval, modern, undated]
    required_per_sub: [context_list, finds_summary, interpretation]
  - id: discussion
    label: "Discussion"
  - id: archive_statement
    label: "Archive Statement"
    required_fields: [repository, accession_categories, deposition_date]
prohibited_terms:
  - "significant" # HE style guide disallows this term
  - "unique"
  - "important finds"
heading_style: "sentence_case"
figure_caption_format: "Fig. {n}: {description}. Scale 1:{scale} unless noted."
citation_style: "author_date"

Compliance Model Configuration
Parameter	Value
Model	unsloth/gemma-4-E2B-it-GGUF (same instance as Phase 2, Q4_K_M)
VRAM footprint	~2.1 GB
Temperature	0.1 — near-deterministic editing
System prompt	Specifies jurisdiction template rules; instructs model to output ONLY the corrected section
Processing mode	Section-by-section (not whole draft at once) — reduces context length and improves precision
Hallucination guard	Model is explicitly forbidden from adding factual claims not present in the Phase 3 draft

Compliance Checks Performed
    • All mandatory sections present and non-empty.
    • All required fields within each section are addressed (model flags missing ones with [MISSING: field_name] placeholder).
    • Executive summary word count within template limit.
    • Prohibited terms replaced with jurisdiction-appropriate alternatives.
    • Heading capitalisation style corrected.
    • Figure captions reformatted to template standard.
    • Section ordering matches template sequence.
    • Context numbers formatted consistently (e.g., square brackets, leading zeros).

ℹ  NOTE	The compliance pass does NOT perform factual review — it cannot know if a context sheet interpretation is archaeologically sound. Its job is structural and stylistic conformity only. The site director must review the draft for factual accuracy before submission.



PHASE
5	Assembly & Export
Compile the refined report with figures, tables, and appendices into final deliverables

Overview
Phase 5 is entirely rule-based — no model inference. It assembles the compliant Markdown sections, embeds figures, generates appendices, and exports the final report in the required formats. All figure numbering, cross-references, and appendix indexing are handled by the assembler.
Assembly Process
    8. Load all Phase 4 compliant Markdown sections in template order.
    9. Resolve figure references: match [FIG:photo_DSC0042] tokens in text to the corresponding processed image in assets/.
    10. Generate appendices automatically: Context Register, Finds Concordance, Sample Register — all from Phase 1 JSON.
    11. Generate bibliography from any citations added during the compliance pass.
    12. Compile final Markdown document.
    13. Export via pandoc: (a) DOCX with heritage org style template, (b) PDF for submission, (c) TEI-XML if required by repository.
    14. Package archive ZIP: final report + all Phase 1-2 data files + original assets + pipeline log.

Export Formats
Output	Format	Tool	Notes
Submission draft	DOCX	pandoc + reference.docx template	Uses organisation-specific reference.docx for correct styles
PDF for submission	PDF	pandoc + LaTeX	Embedded fonts, print-ready
Digital archive	TEI-XML	Custom XSLT	Optional; for national archive repositories
Data archive	ZIP	Python zipfile	All JSON, Markdown, assets, logs
Harris Matrix	SVG	graphviz (dot)	Generated from stratigraphic relationships in context JSON


Command-Line Interface Design
The pipeline is invoked via a single entry-point script. All configuration is handled via a project config file or CLI flags. The interface is designed to be usable by a non-developer site director.

# Initialise a new project
erd init --name 'Stoneyfield Farm 2026' --jurisdiction historic_england_cl3

# Run the full pipeline
erd run --input ./field_records/ --project stoneyfield_2026

# Run a single phase (useful for re-running after manual corrections)
erd run --project stoneyfield_2026 --phase 3

# Run from phase 3 onward (e.g. after editing digitised JSON)
erd run --project stoneyfield_2026 --from-phase 3

# Open the review dashboard (flags items needing human input)
erd review --project stoneyfield_2026

# Export final report
erd export --project stoneyfield_2026 --format docx,pdf

# List available jurisdiction templates
erd templates list

Review Dashboard
The erd review command opens a simple terminal-based (or optional browser-based) review interface. It presents each flagged item — a low-confidence OCR field, a missing mandatory section, a geometry review trigger — one at a time and allows the user to: accept the AI value, type a correction, or mark as 'to revisit'. Confirmed values are written back to the relevant JSON file and the pipeline state is updated. The user can then re-run from the relevant phase.


Error Handling & Fallback Strategy
Error Condition	Detection Point	Behaviour
Input file corrupt / unreadable	Phase 0 triage	Skip file, log warning, continue; halt if mandatory file missing
OCR confidence below threshold	Phase 1 model output	Flag field REVIEW_REQUIRED; use best guess as placeholder
CadQuery execution failure	Phase 2 sandbox	Flag GEOMETRY_FAIL; skip geometry for that drawing; report proceeds without it
Gemma grounding confidence < 0.6	Phase 2 output	Flag photo as LOW_CONFIDENCE; caption still included but marked
Qwen context exceeds KV cache	Phase 3 pre-flight	Switch to chunk-and-merge drafting automatically; log warning
Phase 3 draft missing mandatory section	Phase 4 compliance check	Insert [MISSING: section_id] placeholder; flag for human completion
Prohibited term not removable in context	Phase 4 compliance	Flag for human review rather than force-replace
pandoc export failure	Phase 5	Retain Markdown output; log error with pandoc command for manual retry
VRAM OOM at any phase	CUDA exception	Catch exception; log; reduce batch size by 50% and retry; if still fails, switch to CPU


Testing Strategy
Unit Tests
    • Phase 0: manifest generation correctness; quality flag logic on synthetic images of known blur/skew.
    • Phase 1: OCR output schema validation; checkbox boolean correctness on synthetic forms with known answers.
    • Phase 2: CadQuery script executability rate on test drawing set; caption JSON schema validity.
    • Phase 3: Section label extraction correctness; context reference completeness (all contexts in data appear in draft).
    • Phase 4: Mandatory field presence after compliance pass; prohibited term absence.
    • Phase 5: DOCX export opens without error; PDF generates correctly; ZIP completeness.

Integration Test Dataset
A reference dataset of 3 fully processed past excavations (with known-good published reports) should be assembled for integration testing. The pipeline output is evaluated against the published report on: context sheet capture rate, period identification accuracy, and prohibited term rate. These metrics are reported in the test log.

Acceptance Criteria
Metric	Target
Context sheet field capture rate (vs manual)	> 90% of fields correctly extracted
Period identification match (vs published report)	> 85% agreement
Mandatory section completeness (post Phase 4)	100% (placeholders acceptable)
Prohibited term rate in final output	0%
VRAM peak across all phases	< 5.5 GB (leaving 0.5 GB OS headroom)
Full pipeline runtime (typical 50-context evaluation)	< 25 minutes on target hardware


Known Limitations & Risks
Limitation	Impact	Mitigation
Zero-To-CAD accuracy on schematic field sketches	High — geometry may be incorrect	Hard gate: all geometry flagged REQUIRES_REVIEW; excluded from report body
TrOCR accuracy on unusual handwriting styles	Medium — key fields may be misread	Confidence threshold flags; review dashboard for human correction
Qwen drafting hallucination risk	Medium — model may invent plausible-sounding details	System prompt explicitly instructs: cite only data present in context; review triggers on uncertain language
Jurisdiction templates require maintenance	Low-Medium — heritage standards change	Template versioning; update process documented; community contribution model
Very large sites (500+ contexts)	Low-Medium — may exceed KV cache	Chunk-and-merge fallback; noted in CLI output
Non-English context sheets	Medium — HTR model may underperform	Language detection in Phase 0; alternative HTR models for French, German, Italian can be specified
Specialist finds (numismatics, archaeometallurgy)	Medium — Qwen may lack specialist knowledge	Flag for specialist review; system prompt notes specialist finds categories

Open Questions for v2.1
    • Should the review dashboard be terminal-only (rich/textual) or offer an optional lightweight web UI (FastAPI + HTMX)? The web UI option is more accessible for non-technical site directors but adds a dependency.
    • Is a Harris Matrix auto-generator within scope for v1? The stratigraphic relationships are available in the context JSON; generating the matrix SVG via graphviz is straightforward but will need validation against manually drawn matrices.
    • What is the minimum hardware specification for the CPU-only fallback path? An M-series Mac (unified memory) handles the full pipeline well. A CPU-only x86 path is possible but Phase 3 inference time will be 10-20x longer.
    • Should the pipeline support ARK system (the open-source excavation database) as a direct data input, bypassing Phase 1 OCR entirely for digitally-recorded sites? This would be a significant time saving.
    • Qwen3.5-4B in Thinking Mode produces a reasoning chain of 1,000–3,000 tokens before the draft. Should this be presented to the site director as a 'confidence log', or silently discarded after logging? User research needed.
