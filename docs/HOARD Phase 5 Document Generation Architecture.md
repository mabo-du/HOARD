# **HOARD Phase 5 Document Generation Architecture**

The final assembly and export stage of the Heritage Observation And Report Drafter (HOARD) presents a highly complex computational challenge. Bridging the gap between structured, machine-readable archaeological data synthesized in earlier phases and publication-ready, legally compliant grey literature requires an architecture that is deterministic, highly scalable, and completely independent of external web services. Operating under the strict hardware constraints of a fully local, CPU-bound execution environment on consumer-grade machines, the Phase 5 architecture must synthesize programmatic document generation, geometric data rendering, and archival-grade file formatting. This report outlines the definitive architectural blueprint, specialized data models, and algorithmic strategies required to implement this system efficiently, accommodating varying international heritage standards without compromising the integrity of the archaeological record.

## **Recommended Pipeline Architecture**

The transformation of Markdown containing YAML frontmatter into distinct output formats—an editable DOCX draft for principal investigator review, an HTML preview for rapid visualization, and a strictly compliant PDF/A-2b archival document for permanent deposition—demands a hybrid pipeline. Relying exclusively on a single monolithic tool introduces unacceptable operational compromises. For instance, while Pandoc is an exceptional baseline format converter, it lacks the fine-grained layout controls required for complex archaeological elements such as multi-page landscape tables or tightly packed photographic plates.1 Similarly, procedural engines like ReportLab offer pixel-perfect control but impose an exceptionally steep learning curve and verbose codebase that hinders the rapid deployment of new jurisdictional templates. Conversely, libraries such as python-docx excel at populating branded templates but entirely lack native, reliable PDF output capabilities.3  
The optimal architecture for HOARD Phase 5 is a hybrid Markdown-Jinja-WeasyPrint pipeline. This approach cleanly decouples content parsing, word processing synthesis, and archival rendering into distinct, purpose-built operational layers that execute sequentially and deterministically within the local Python 3.12+ environment.

### **Architectural Dependency Chain**

The architecture relies on a highly specific sequence of data transformations and specialized Python libraries.

| Operational Stage | Primary Technology | Python Dependency | Function within Pipeline |
| :---- | :---- | :---- | :---- |
| **Frontmatter Extraction** | YAML Parsing | pyyaml | Ingests metadata and structured frontmatter from Phase 4, converting it into a global context dictionary for template injection. |
| **HTML Preview Generation** | Pandoc | pypandoc | Converts the base Markdown body into semantic HTML5, applying basic CSS for rapid local browser preview. |
| **DOCX Template Rendering** | Jinja2 & Word XML | docxtpl | Loads a branded, jurisdiction-specific .docx template and populates it with the context dictionary and Markdown-converted sub-documents.3 |

| **Cryptographic Signing** | PAdES Injection | pyhanko | Applies offline digital signatures to the finalized PDF/A byte stream to guarantee document authenticity.9 |

### **Data Flow and Interleaved Assets**

The pipeline initiates by extracting the YAML frontmatter generated by the Phase 4 compliance refinement engine. This metadata—comprising site codes, heritage identifier numbers, author details, and chronological data—is immediately mapped into a unified Python dictionary. This dictionary acts as the definitive global context variable passed to the downstream templating engines.  
For the generation of the editable DOCX draft, the architecture utilizes the docxtpl library, which embeds a Jinja2 templating engine directly into Microsoft Word documents.3 A designated jurisdiction template containing pre-formatted Jinja2 tags is loaded into system memory. As the engine evaluates the template, it dynamically unpacks the context dictionary. When the parser encounters an interleaved figure or table reference within the Phase 4 Markdown body, the Python orchestration script intercepts the reference. For tables, the raw CSV or JSON data is processed via pandas and injected into the document using docxtpl's native dynamic table row generation tags.13 For figures, the script verifies the existence of the image asset in the local directory and embeds it with appropriate scaling parameters.  
If a referenced figure or table is missing from the local directory structure, the architecture implements a non-blocking error handling strategy. The execution does not halt; instead, the Jinja2 rendering logic replaces the missing asset with a standardized SVG placeholder indicating the missing filename and logs a critical warning to the Phase 5 output console. This ensures the principal investigator receives a complete, albeit flagged, draft rather than a fatal pipeline crash.

### **Frontmatter to Document Property Mapping**

The translation of YAML frontmatter into both Microsoft Word built-in properties and PDF/A XMP metadata profiles is a strict requirement for digital archiving. The mapping strategy emphasizes the Dublin Core metadata standard, which is heavily favored by the Archaeology Data Service (ADS) and the Forum on Information Standards in Heritage (FISH).1

### **Template Fallback Strategy**

To manage the high variability of regional reporting requirements, HOARD implements a hierarchical template resolution algorithm. When a document generation request is triggered, the system searches for formatting templates using a strict fallback chain. It first queries the precise jurisdiction and version requested (e.g., templates/England/CL3/2026/). If a specific template is unavailable, the system falls back to a regional base template (e.g., templates/England/base/), and finally to a globally standard archaeological template. This guarantees that the pipeline will always produce a structurally sound document, even if the highly specific jurisdictional branding is absent.

### **Architecture Rationale**

The Hybrid Markdown-Jinja-WeasyPrint pipeline is the optimal architecture for HOARD Phase 5 because it perfectly balances the need for complex, branded layouts with the strict requirement for offline, CPU-only PDF/A generation. Relying solely on python-docx is insufficient as it cannot natively produce the archival PDF formats mandated by heritage authorities without relying on external, API-driven conversion tools like Microsoft Word automation, which violates the local execution constraint. Conversely, utilizing Pandoc exclusively limits the ability to construct highly customized, visually dense artifact plates and specialized multi-page tables, as Pandoc's intermediate LaTeX or HTML parsing often struggles with absolute positioning. By employing docxtpl for the DOCX generation, HOARD leverages Jinja2 to programmatically populate meticulously branded Word templates, ensuring principal investigators receive highly editable, professional drafts. Simultaneously, the parallel use of WeasyPrint converts the semantic HTML representation of the report directly into a compliant PDF/A-2b document.8 WeasyPrint's native support for embedding color profiles, subsetting fonts, and injecting Dublin Core metadata directly into the XMP dictionaries ensures that the final archival output strictly adheres to the stringent long-term preservation requirements of digital repositories like the Archaeology Data Service.

## **Multi-Jurisdiction Template Engine Architecture**

Supporting an expansive array of jurisdictions—encompassing over 50 individual US states, various European national standards, and specific provincial guidelines in Canada and Australia—necessitates a robust template architecture that strictly isolates content logic from presentation rules. Hardcoded templates, such as those historically utilized by proprietary CRM tools like Wildnote, frequently suffer from user complaints regarding extreme formatting rigidity and the inability to adapt to hyper-local variations. State-level systems display wide variance: the Oregon State Historic Preservation Office utilizes web-portal direct entry, Texas relies on Access database exports converted to CSV, and California's DPR 523 forms are highly structured documents often generated via procedural PDF engines.16 To navigate this fragmentation, HOARD requires a dual-layered template system.

### **Isolating Content and Layout Templates**

The core architectural tenet of the multi-jurisdiction engine is the total bifurcation of templates into two distinct categories: Content Templates and Layout Templates.  
Content Templates dictate the required data elements, the specific sequence of stratigraphy discussion, the mandatory inclusion of executive summaries, and the citation style (e.g., Society for American Archaeology format versus Harvard). These are defined strictly via JSON Schema and are processed exclusively during Phase 4\.17 The Phase 4 Large Language Model (LLM) agents validate their structural output against this schema to ensure the flow of the text complies with the jurisdiction's intellectual requirements. If the Phase 4 output fails to generate a schema-mandated section, a validation hook forces a regeneration loop.  
Layout Templates, conversely, are processed in Phase 5 and govern the visual presentation. These are defined via docxtpl Word documents and associated CSS stylesheets. They determine margins, typography, pagination rules, title page branding, and header/footer configurations.3 This separation ensures that an update to a state's font requirement does not necessitate a fundamental retraining or prompt adjustment of the Phase 4 reasoning engine.

### **The JSON Schema Specification Model**

Each supported jurisdiction is represented by a version-controlled directory containing a schema.json file defining the report's ontology. This schema validates the presence and data types of required sections and metadata fields. For instance, the Ontario ERO 026-0216 standard demands highly specific subdivisions for the "Record of Finds." The JSON Schema enforces this tree structure hierarchically. By expressing the jurisdiction's requirements as a JSON Schema, Phase 4 can deterministically validate the report before Phase 5 is permitted to execute, ensuring that the final rendering engine does not crash due to missing mandatory variables.  
Version management for these templates is critical, as standards evolve continuously (e.g., the Historic England 2026 update, the Netherlands KNA 2026 Leidraad update). Templates must utilize Semantic Versioning (SemVer). The JSON descriptor file contains validity epochs, such as valid\_from: "2026-01-01" and valid\_until: "2030-12-31". When the pipeline initializes, it checks the current system date against the template's validity epoch. If a user attempts to generate a report using an outdated template, the orchestration script injects a prominent watermark into the draft DOCX and logs a console warning alerting the user that the standard is deprecated, ensuring compliance with the most current heritage laws.

### **Community Contribution Workflow**

To scale across hundreds of local jurisdictions globally, HOARD must adopt a decentralized, open-source template repository model. A unified templating API will allow CRM professionals and heritage officers to contribute new jurisdiction parameters without writing core Python code. The contribution workflow requires the submission of a single archive package containing four core files: a schema.json to define the content rules, a template.docx containing the Word layout with Jinja2 tags, a style.css file dictating the WeasyPrint PDF layout, and a metadata.yml file defining the jurisdiction name, version, and the contributor's details. Upon local deployment, the HOARD Phase 5 engine automatically indexes these packages, dynamically expanding the available jurisdiction options in the user interface.

## **Specialist Appendix Generation**

In commercial archaeology, particularly within the UK under the Chartered Institute for Archaeologists (CIfA) guidelines, a final report must include highly structured specialist appendices adhering to the Toolkit for Specialist Reporting.19 These appendices are categorized into three distinct tiers: Type 1 (Description) for basic quantification of relatively insignificant assemblages; Type 2 (Appraisal) for post-excavation assessments evaluating preservation and chronological coherency; and Type 3 (Full Analysis) representing deep, integrated interpretation.19 Generating these appendices directly from raw, structured tabular data—such as finds catalogs and environmental sample results—without introducing LLM hallucinations is a critical requirement of the Phase 5 architecture.

### **Data Models for Archaeological Materials**

Existing archaeological recording systems, such as the Integrated Archaeological Database (IADB), ARK, and Intrasis, utilize highly relational, object-oriented SQL schemas to manage complex finds data.22 To operate optimally within HOARD's offline, lightweight framework, the system abstracts these complex relational models into an in-memory dataframe strategy utilizing pandas, mapping standardized CSV exports to structured schemas prior to document insertion.  
The data models for each specialism dictate the requisite fields necessary for aggregation.

| Specialism | Primary Aggregation Metrics | Morphological/Descriptive Fields | Taphonomic/Condition Fields |
| :---- | :---- | :---- | :---- |
| **Pottery** | Count, Weight, Estimated Vessel Equivalent (EVE), Minimum Number of Individuals (MNI) 26 | Fabric type, form, rim diameter, decoration | Abrasion, burning, residue presence |
| **Animal Bone** | Number of Identified Specimens (NISP), Minimum Number of Elements (MNE) 27 | Taxon (species), element (e.g., femur), side, age/fusion | Butchery marks, carnivore/rodent gnawing, weathering 29 |
| **Lithics** | Count, Weight | Raw material, technology, tool type, retouch type | Patination, edge damage, burning 16 |
| **Environmental** | Initial sample volume, processed flot volume 30 | Plant remains (charred seeds), charcoal quantification | Preservation status, mineral replacement 32 |

For pottery, the data model requires hierarchical grouping: Context \-\> Fabric Type \-\> Form. The calculation of Estimated Vessel Equivalent (EVE)—a vital statistical metric used to eliminate the unpredictable biases inherent in simple sherd counting—must be highly deterministic, relying on the summation of surviving rim percentages.33 The pipeline utilizes pandas.groupby() functions to calculate these metrics mathematically before inserting the summary tables into the Jinja2 context variables.  
For environmental samples processed via flotation, the schema must distinctly separate the 'flot' (the light fraction containing floating organic material such as seeds and charcoal) from the 'heavy residue' (non-floating material such as micro-debitage and fish scales).30 The data model accommodates these distinct fractions, calculating concentration densities by dividing the quantified remains by the initial sample volume, enabling Phase 5 to generate standardized archaeobotanical summary tables.

### **Controlled Vocabularies and Ontology Integration**

To guarantee interoperability and compliance with regional data standards, the data ingestion parameters must align with established thesauri. The architecture integrates the Forum on Information Standards in Heritage (FISH) Archaeological Objects Thesaurus and the Getty Art & Architecture Thesaurus (AAT).35 Rather than relying on fuzzy LLM matching during text generation, Phase 0 and Phase 1 perform entity resolution against these structured vocabularies. Phase 5 strictly outputs the resolved terminology. This ensures that regional synonyms—such as identifying an object as a "potsherd" versus a "pottery fragment"—are normalized according to the selected jurisdiction's specific vocabulary requirements, preventing compliance failures during final archival deposition.

### **Prose Generation Approach: Deterministic NLG vs. LLM**

The strategy for generating the prose surrounding these specialist appendices requires a highly nuanced approach to prevent data hallucination. Relying on an autoregressive LLM to summarize tabular specialist data introduces an unacceptable risk of mathematical hallucination (e.g., an LLM stating a pottery count is 127 when the underlying table lists 124).  
For Type 1 (Description) appendices, which consist entirely of basic quantification and standardized material descriptions, Phase 5 avoids LLM prose generation entirely. Instead, the architecture utilizes Rule-Based Natural Language Generation (NLG) executing directly through the Jinja2 template engine.3 The logic leverages the mathematically processed pandas dataframes to populate highly constrained syntactic templates. A Jinja2 template snippet operates as follows: The pottery assemblage comprised {{ total\_count }} sherds weighing {{ total\_weight\_kg }} kg, with an EVE of {{ total\_eve }}. The fabric was predominantly {{ dominant\_fabric\_name }} ({{ dominant\_fabric\_percentage }}%). By passing the aggregated dictionaries into this syntax, the output reads as fluid, professional prose that is mathematically guaranteed to match the appended data tables perfectly.  
For Type 3 (Full Analysis) appendices, which require deep interpretation integrated with site phasing, deterministic templates are insufficiently flexible. However, fine-tuning specialized miniature text generation models for each distinct specialism (pottery, bone, lithics) would dramatically exceed the 6 GB VRAM constraint of the target consumer hardware. Therefore, the optimal strategy relies on advanced prompt engineering applied to the main Phase 3 LLM. The Phase 3 model is provided with the pre-aggregated summary statistics and prompted to generate purely interpretative paragraphs focusing on site economy, chronological implications, and regional context. Phase 5 then sequentially injects the deterministic mathematical summaries adjacently to these interpretative paragraphs, seamlessly blending secure, generative insights with mathematically infallible facts.

## **Figure and Plate Auto-Layout Strategy**

Archaeological reports are fundamentally visual documents, relying heavily on site location maps, spatial reconstructions, and intricate artifact illustration plates. The manual composition, scaling, and alignment of these visual elements represent a primary bottleneck in commercial post-excavation workflows. HOARD Phase 5 automates this process through a sophisticated combination of geometric bin packing algorithms and dynamic procedural data rendering.

### **Algorithmic Bin-Packing for Photographic and Artifact Plates**

To compose photographic plates and artifact illustrations automatically, the architecture utilizes the two-dimensional knapsack problem methodology via the Python library rectpack.40 This heuristic algorithm effectively packs a series of irregular bounding boxes (the images) into the smallest number of restricted "bins" (the standard A4 or US Letter page dimensions).  
The objective function of the algorithm seeks to maximize the filled area of the page bounds while strictly adhering to the constraint of non-overlapping geometries.43 For archaeological artifact illustrations, particularly pottery profiles and lithic drawings, consistent scaling is an absolute imperative.45 The images are pre-processed by the Phase 2 spatial engine to standardize their Dots Per Inch (DPI) to a known physical scale. The rectpack algorithm is fed the absolute pixel dimensions of each standardized image. Margins are dynamically added as a buffer (configured via the bx, by parameters in rectpack) to accommodate necessary white space and to provision room for descriptive captions and figure identifiers directly beneath each packed element.42 The optimized output coordinates generated by rectpack are then utilized by the Python Imaging Library (Pillow) to render the final composite plate image before insertion into the DOCX or PDF document stream.

### **Procedural Generation of Figures from Tabular Data**

Certain complex archaeological figures are not ingested as raster images but must be procedurally generated directly from the raw tabular data during the Phase 5 execution to ensure absolute data fidelity.  
The generation of Harris Matrix diagrams is a primary example. The stratigraphic sequence of the site is parsed from the observations.csv format, strictly conforming to the Harris Matrix Data Package specification.17 The relative chronologies of context nodes—specifically the earlier-later relationships—are ingested into the pipeline. To visualize this data, the pipeline utilizes the Graphviz library, rendering a Directed Acyclic Graph (DAG) utilizing the DOT language.49 Crucially, the rendering engine is constrained by the "ortho" layout attribute.48 This forces the algorithm to draw edges as orthogonal polylines, ensuring that the visual output directly mirrors the conventional, rigid, top-down rectilinear structure expected by archaeologists, rather than an organic, curved network graph.52  
Specialist data charts require entirely different procedural approaches. Environmental data, particularly palynological (pollen) species counts, require complex, multi-axis stratigraphic representations. The pipeline incorporates matplotlib and specialized stratigraphic wrappers such as psyplot or the psy-strat extension to render high-resolution pollen diagrams.53 The vertical axis represents deposit depth or radiocarbon chronology, while the horizontal distributions represent species frequency percentages.55 Faunal NISP distribution charts and pottery fabric chronotype graphs are similarly rendered dynamically, ensuring that the graphics update instantaneously and accurately if the underlying source data is modified during the review process.

### **Embedded vs. Appendix Layouts and Resolution Management**

Jurisdictional templates dictate specific placement rules regarding whether figures are interleaved contextually within the body text or aggregated collectively as plates at the end of the document. The Jinja2 logic parses this rule from the template definition and outputs the image reference anchors accordingly.  
Image resolution management is enforced strictly to balance visual fidelity against total file size constraints. Phase 5 utilizes a dual-profile generation strategy. For the web-preview and draft DOCX profiles, images are downsampled to 150 DPI, utilizing aggressive JPEG compression with quality optimization enabled (e.g., jpeg\_quality=60 within the WeasyPrint parameters).8 For the archival PDF/A profile, images are retained at a high-resolution 300 DPI utilizing lossless or high-quality formats. To prevent interpolation artifacts that are heavily penalized by PDF/A archival standards, CSS image-rendering properties such as image-rendering: crisp-edges are programmatically applied to rasterized technical drawings to strictly avoid anti-aliasing blurring.56 Figure numbering is managed autonomously via a global sequence counter within the Jinja2 environment, guaranteeing that cross-references in the body text automatically and accurately map to the generated layout.

## **PDF/A Compliance and Digital Archiving**

The deposition of digital grey literature into trusted repositories such as the Archaeology Data Service (ADS) in the UK or state-level Historic Preservation Offices requires rigorous adherence to long-term archiving standards, most notably the PDF/A format.57 Failure to meet these technical standards results in immediate rejection by the repository.

### **PDF/A Generation Mechanics and Compliance Checklist**

The WeasyPrint engine provides native, robust support for generating PDF/A and PDF/UA formats.15 For the HOARD pipeline, the specific target standard is PDF/A-2b. This standard, based on the ISO 32000-1 specification, is selected over the older PDF/A-1b because it permits superior handling of transparent image elements—which are absolutely vital for overlaid GIS site maps and spatial reconstructions—and generally results in smaller overall file sizes.8  
To achieve PDF/A-2b compliance without requiring a two-pass Ghostscript conversion pipeline, Phase 5 executes a highly constrained single-pass rendering process passing specific arguments to the Python WeasyPrint API:

1. **Variant Specification:** The output variant is explicitly defined using the parameter pdf\_variant="pdf/a-2b".8  
2. **Strict Font Embedding:** All fonts must be fully embedded and legally embeddable for universal rendering without relying on the end-user's local system fonts. WeasyPrint utilizes the hb-subset (HarfBuzz) library to automatically embed and subset OpenType and TrueType fonts, fulfilling this critical requirement seamlessly.15  
3. **Color Space Normalization:** Color spaces must be defined in a device-independent manner. The pipeline injects the \--srgb flag to forcibly bind a standard sRGB color profile to the document, ensuring images render consistently across varying archival platforms over the coming decades.8  
4. **Element Sanitization:** Forbidden elements—such as embedded audio, video, Javascript, and external absolute links that might suffer link rot—are programmatically stripped from the HTML source by a custom Python pre-processor before the document reaches the WeasyPrint renderer.62

### **Metadata Embedding and Integration Strategy**

Digital repositories mandate the inclusion of robust technical and descriptive metadata.59 The Phase 5 pipeline maps the YAML frontmatter into custom HTML meta tags within the intermediate semantic structure. When executing WeasyPrint, the \--custom-metadata flag embeds these parameters deep within the PDF's internal XMP metadata dictionary.8 This metadata explicitly aligns with the MIDAS Heritage standard, ensuring a base level of interoperability for recording heritage assets.14  
Regarding integration with submission portals such as OASIS, direct API integration for third-party tools is not highly standardized, and attempting external API calls violates HOARD's strictly local execution constraints. Therefore, manual upload by the user remains the standard practice. HOARD's role is to automatically generate a perfectly compliant submission package—comprising the PDF/A-2b report and a supplementary CSV metadata sheet—and to automatically enforce the strict file naming conventions dictated by the repository (e.g., formatting the output strictly as {oasis\_id}\_{site\_code}\_report.pdf), thereby drastically reducing the administrative burden on the depositor.

### **Digital Signatures and PAdES Authentication**

While not universally required by the ADS, an increasing number of commercial contracts, municipal planning authorities, and federal agencies require cryptographic digital signatures to verify the legal authenticity of the final archaeological report. To achieve this entirely offline, HOARD incorporates the pyHanko library to append PAdES (PDF Advanced Electronic Signatures) directly to the rendered PDF/A document.9  
PyHanko operates directly on the raw byte stream of the generated PDF file. It supports the baseline PAdES-B and PAdES-LTV (Long Term Validation) profiles, allowing the attachment of both visible and invisible digital signatures utilizing local cryptographic keys stored on the user's hardware.10 By injecting the signature data directly into the PDF's cryptographic dictionaries 66, HOARD eliminates any reliance on external validation services like Adobe Acrobat, preserving the core philosophy that zero project data leaves the local machine during generation.

## **Performance and Scale Operations**

A primary technical obstacle for CPU-based document generation pipelines is the extreme memory bloat associated with rendering exceptionally long documents. While rural evaluation reports may span a mere 30 pages, commercial archaeology reports for major infrastructure projects (such as HS2 or Crossrail in the UK) frequently exceed 200 pages and incorporate thousands of rows of dense tabular data.

### **Benchmarks and Memory Management**

WeasyPrint relies heavily on system RAM during the CSS cascade and pagination calculation phases. Generating a 200-page document comprising extensive HTML tables can easily consume upwards of 2.6 GB of RAM, putting significant strain on a 16 GB machine that is simultaneously shared with desktop CRM applications and GIS software.68 While this fits within general operational bounds, optimization is mandatory to prevent out-of-memory (OOM) crashes during concurrent generations.  
To mitigate this bottleneck, the architecture implements several aggressive memory conservation tactics. First, HTML whitespace minification is applied. Removing superfluous whitespace and line breaks from the intermediate HTML drastically reduces the Document Object Model (DOM) tree size, demonstrating a highly measurable memory footprint reduction during the parse phase.68 Second, the pipeline enforces strict disk caching for assets. By default, WeasyPrint stores all parsed images directly in RAM. Phase 5 overrides this behavior by setting a local \--cache-folder, forcing the engine to flush high-resolution artifact plates to the solid-state drive during generation, exchanging a marginal I/O speed penalty for massive RAM stabilization.8 Finally, complex tables spanning multiple pages trigger recursive, highly expensive pagination calculations. Phase 5 pre-calculates table chunking via pandas, manually splitting excessively long context catalogues into smaller, discrete tables that WeasyPrint can cascade with linear time complexity.56

### **Parallel Generation and Document Assembly**

To dramatically accelerate generation times for extensive reports, the pipeline employs a multi-threaded document chunking strategy. The overall report is programmatically divided into logical chapters (e.g., Introduction, Stratigraphic Sequence, Specialist Appendices). Each chapter is generated in parallel across the CPU's available processing cores as independent, temporary DOCX sub-documents. Once complete, the docxcompose library is utilized in a rapid consolidation step to append the temporary documents sequentially into the final master DOCX file.4 This circumvents the exponential time-complexity curve associated with a single process attempting to parse a massive XML tree. For PDF output, figures and procedural graphics are generated via multiprocessing threads simultaneously, caching the resultant image files to disk just before the main WeasyPrint sequence calls them for final rendering.

### **File Size Constraints and Optimization**

Digital filing systems impose strict file size limits; the Advisory Council on Historic Preservation (ACHP) in the US imposes a 20MB e-filing limit, while the ADS typically restricts standard deposits to 50MB. HOARD employs algorithmic size management via the optimize\_images=True parameter within WeasyPrint.8 If the final output file exceeds the user-defined threshold, the pipeline initiates an automatic recursion loop, stepping down the embedded image DPI and increasing JPEG compression dynamically until the file falls within the accepted size bounds. This ensures the output is immediately ready for online submission without requiring the user to rely on unstable third-party PDF compression tools that frequently strip out vital PDF/A metadata.

To deploy Phase 5 effectively and securely, engineering efforts must be structured incrementally, prioritizing core compliant document outputs before layering complex automated spatial rendering algorithms.

| **Stage 3: Archival Compliance** | Embed WeasyPrint with strict PDF/A-2b flags, HarfBuzz font subsetting, and sRGB color profiles. Implement Dublin Core XMP metadata writing. Construct the JSON Schema validator for Phase 4 Markdown enforcement. | **Core** |

This comprehensive architecture guarantees that HOARD Phase 5 achieves a rigorous operational balance: maintaining total offline data privacy, ensuring rigid adherence to complex international heritage taxonomies, and delivering archival-quality, legally compliant outputs efficiently on standard consumer hardware.
