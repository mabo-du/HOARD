# From CLI to Click: A Strategic Roadmap for Making the HOARD Archaeological Pipeline Accessible to Non-Terminal Users

This research report provides a comprehensive evaluation of user interface (UI) accessibility strategies for HOARD, a command-line interface (CLI)-based archaeological reporting pipeline. The central objective is to make HOARD usable for non-terminal commercial archaeologists on Windows and macOS, who manage small teams and rely on software like QGIS and Excel but have no experience with terminal-based tools . The analysis rigorously assesses seven proposed approaches—CLI progressive disclosure, Textual TUI enhancement, local web wrapper, Tauri desktop shell, Trowel integration, configuration presets, and guided onboarding—against a strict set of developer-centric constraints. These constraints include offline-first operation, minimal core dependencies, maintainability by a single developer, secure credential handling, cross-platform support, and genuine solvability of the cold-start problem. The report culminates in a ranked recommendation of the most effective strategies, providing deep-dive analyses of the top two options and a strategic synthesis for implementation.

## Foundational Analysis: The Dual Barriers of Installation and Interface

The challenge of making HOARD accessible to non-terminal users is not a singular problem but a sequence of two distinct and formidable barriers that must be addressed: the "cold-start" installation barrier and the subsequent interface adoption barrier. Understanding these two layers is fundamental to evaluating any proposed solution effectively. The target user persona—a commercial field archaeologist—is defined by their comfort with graphical software like QGIS and CRM systems but an outright avoidance of the command line . This dual-barrier framework dictates that any successful strategy must solve both problems in a coherent, low-maintenance manner.

The first and most critical barrier is the cold-start problem, which encompasses the entire process of getting the application installed and running for the very first time. For a user unfamiliar with Python ecosystems, the traditional installation method—requiring a pre-existing Python installation, adding it to the system PATH, managing virtual environments, and executing `pip install hoard-erd`—is a significant and likely insurmountable hurdle . This initial friction point represents the first failure before any graphical interface can even be presented. Any solution that presumes the user can navigate these steps is fundamentally flawed. The developer has explicitly identified this as a primary axis of evaluation, demanding that any recommended approach must work for a user who has never opened a terminal . The resolution of this installation problem is therefore not merely a prerequisite; it is the foundational requirement upon which all other accessibility efforts depend. The correct path forward, as clarified by the developer, involves leveraging modern tooling like `uv`, a single-binary Rust-based Python installer, to collapse the multi-step installation chain into a simple, one-liner command that even a novice user can execute [[11](https://discuss.python.org/t/uv-another-rust-tool-written-to-replace-pip/46039), [54](https://dev.to/epam_india_python/meet-uv-the-next-gen-python-package-manager-built-for-speed-and-simplicity-3og7)]. Following installation, the second part of the cold-start problem—the "running" problem—must be solved. This asks how a user can start the application without opening a terminal themselves. The proposed solution is elegant in its simplicity: upon first initialization (`hoard init`), the application writes a platform-appropriate launcher script to the desktop (a `.bat` file on Windows or a `.command` file on macOS) that encapsulates the necessary command to start the application server and open a browser [[51](https://stackoverflow.com/questions/62097824/running-a-python-script-by-double-clicking-desktop-icon-on-mac-os), [52](https://superuser.com/questions/966946/how-to-run-python-script-in-a-terminal-window-by-double-clicking-it)]. This transforms the act of starting the application into a simple double-click, abstracting the underlying terminal commands entirely. This approach deliberately rejects more complex solutions like background services, which introduce high maintenance overhead and security risks, or bespoke OS installers, which require per-platform CI/CD pipelines and constant maintenance [[2](https://stackoverflow.com/questions/32404/how-do-you-run-a-python-script-as-a-service-in-windows), [3](https://discuss.python.org/t/running-python-scripts-at-startup-and-in-background-launchd-macos/79855)].

Once the cold-start barrier is overcome, the second major obstacle emerges: interface adoption. Even if a user successfully navigates the installation, they are described as being intimidated by the terminal environment . They require a graphical representation of the workflow that abstracts away the complexity of individual CLI commands. Their needs span the entire project lifecycle: creating a new project, ingesting raw field records (e.g., context sheets, photographs), initiating and monitoring the six-phase AI-assisted pipeline, reviewing flagged items, and exporting the final publication-ready reports. The existing UI within HOARD consists of a Rich-based dashboard that appears *after* each pipeline phase for review purposes . While valuable, this does not address the initial setup, project creation, or the execution of the pipeline phases themselves. Therefore, the goal is to find an interface strategy that lowers the cognitive load associated with navigating this multi-stage process without introducing a divergent codebase that becomes a maintenance burden for a solo developer .

A crucial factor that significantly alters the accessibility calculus is the existence of the "Ultra-light" hardware tier within HOARD's current architecture . This tier allows the full six-phase pipeline to run using only cloud-based AI providers like OpenAI, Anthropic Claude, and Google Gemini, completely bypassing the need for a local GPU or the Ollama framework . This feature is a powerful accessibility lever because it removes the most significant technical barrier—hardware requirements—for a non-technical Windows or macOS user. It implies that a user with basic technical literacy (the ability to install `uv` and run a launcher script) and API keys for a cloud provider could, in theory, run the entire pipeline today . The primary obstacles shift from being purely technical (lacking a GPU) to procedural and navigational (how to set up a project, where to put files, what the different phases do). This insight reframes the problem: the path to usability is less about overcoming a high-tech threshold and more about designing an intuitive user experience. Consequently, any recommended UI strategy must be considered in tandem with a substantial investment in guided onboarding, clear documentation, and simplified configuration presets. The `uv` + launcher script solution addresses the technical installation hurdle, leaving the procedural/navigational hurdle as the primary focus for any new UI intervention . This foundational understanding—that the main challenge is now UX, not tech—guides the evaluation of all proposed strategies, prioritizing those that offer the greatest improvement in user guidance and workflow clarity with the least increase in maintenance burden.

## Comparative Evaluation of Proposed Accessibility Strategies

To address the dual barriers of installation and interface, seven distinct strategies were proposed. Each was evaluated against a weighted rubric designed for a solo developer with limited maintenance resources, considering criteria such as solo maintenance burden, effort to MVP, cold-start solvability, dependency impact, reach, security risk, and synergy with existing tools . The following table presents the final ranked evaluation of these strategies, followed by a detailed rationale for each position.

| Rank | Strategy | One-Line Rationale |
| :--- | :--- | :--- |
| 1 | **Local Web Wrapper (FastAPI)** | Offers broadest reach to non-terminal users, solves cold-start via `uv`+launcher scripts, and maintains a lean architecture with minimal core dependency impact. |
| 2 | **Tauri Desktop Shell** | Provides a native, polished user experience and fully solves cold-start, leveraging existing developer expertise. Higher maintenance risk due to a hybrid architecture. |
| 3 | **Guided Onboarding Flow** | Low-effort, high-leverage intervention that directly addresses user confusion and procedural friction without adding new dependencies or a separate UI. |
| 4 | **Configuration Presets** | A low-risk, low-effort way to reduce cognitive load and standardize setups, complementing other accessibility improvements. |
| 5 | **Progressive Disclosure (CLI Wizard)** | Enhances the familiar CLI environment, guiding users through complex workflows. Less impactful than a full GUI but easier to implement than TUI/web solutions. |
| 6 | **Trowel Integration** | High synergy with an existing tool in the ecosystem, but functionally incomplete as it can only act as a downstream editor, not a pipeline entry point. |
| 7 | **Textual TUI Enhancement** | Narrowest potential audience among all options; its value is limited to a niche group of users who tolerate terminals but are overwhelmed by raw CLI output. |

The **Local Web Wrapper (FastAPI)** ranks first due to its exceptional balance of user reach, maintainability, and alignment with core constraints. It targets the primary user segment—those who avoid terminals—by providing a familiar, browser-based interface . Critically, it integrates seamlessly with the proposed cold-start solution: `uv` handles the Python package installation, and a simple launcher script starts the FastAPI server locally at `http://localhost:8765` [[40](https://fastapi.tiangolo.com/)]. This approach keeps the core HOARD package lightweight, as the web components would be an optional extra. The choice of frontend technology offers flexibility; HTMX is particularly compelling as it enables building a reactive UI with server-side rendering, minimizing frontend complexity and build steps [[43](https://blakecrosley.com/guides/fastapi-htmx)]. The main risk lies in securing the local server and ensuring safe handling of the credential vault unlock process, but this is a manageable engineering challenge [[127](https://fastapi.tiangolo.com/tutorial/security/)].

**Tauri Desktop Shell** secures the second position, primarily because of the developer's existing experience with Tauri 2 from the StratiGraph project . This reduces the learning curve and speeds up development. A Tauri app bundles the application into a native executable for Windows (.exe) and macOS (.dmg), completely eliminating the cold-start problem for the end-user [[62](https://dev.to/ottoaria/tauri-in-2026-build-cross-platform-desktop-apps-with-web-technologies-better-than-electron-11mo)]. The user simply runs the installer and clicks an icon. This provides the most seamless and "app-like" experience. However, this comes with higher architectural complexity. The Tauri shell (Rust and web technologies) must communicate with the Python CLI, requiring careful IPC management [[16](https://lobehub.com/skills/neversight-skills_feed-understanding-tauri-ipc)]. There is also a risk of version drift between the Tauri shell and the Python backend, necessitating robust release coordination. While highly desirable from a user-experience standpoint, the maintenance burden is greater than the web wrapper option.

**Guided Onboarding Flow** is ranked third as a high-leverage, low-effort baseline improvement. Before any new UI is built, improving the initial user experience is paramount. An interactive `hoard wizard` command, combined with a `hoard doctor` command to check prerequisites and offer fixes, directly addresses the user's need for step-by-step guidance . This can be implemented entirely within the existing CLI framework. The welcome screen on first launch that suggests running the wizard further lowers the barrier to entry . This strategy requires no new dependencies and minimal code changes, yet it has the potential to drastically reduce the number of confused users who abandon the tool prematurely. It is a necessary foundation upon which more advanced UIs can be built.

**Configuration Presets** follow in fourth place. This strategy focuses on reducing cognitive load by simplifying the setup process. Commands like `hoard init --quick` or `hoard init --template heritage-survey` allow users to bootstrap a project with minimal input, relying on smart defaults for jurisdictions, hardware profiles, and model settings . These presets can be shared as simple JSON files, enabling consistency across teams . This is a low-risk, low-effort enhancement that complements all other accessibility strategies. It doesn't replace them but makes them more effective by streamlining the initial configuration.

**Progressive Disclosure**, realized through an interactive `hoard wizard`, is ranked fifth. This approach enhances the familiar terminal environment rather than replacing it. By stepping the user through setup, project initialization, and pipeline execution with clear prompts and explanations, it acts as "training wheels" for the CLI . While it doesn't provide a full graphical interface, it significantly lowers the barrier for users who are already comfortable enough with a terminal to use a CLI but are intimidated by its raw output. Its implementation is simpler than a full TUI or web wrapper, but its impact is more limited as it still requires the user to remain within the terminal window.

**Trowel Integration** is ranked sixth. This option benefits from extremely high synergy; bidirectional JSON import/export and shared templates are already implemented, meaning the core data layer is complete . However, its viability is severely hampered by workflow limitations. Trowel is a report drafter that operates on HOARD's *output* (the generated report drafts), not its input (raw field records) . Therefore, it cannot serve as the primary entry point for a new project. The most it can become is a "Run HOARD Phase X" button inside Trowel, which is useful for experienced users who are already deep in their workflow but does nothing to help a new user get started. It is a powerful secondary enhancement but a poor primary solution for the cold-start problem.

Finally, the **Textual TUI Enhancement** is ranked last. While Textual is a modern, web-developer-conceptually-familiar framework for building beautiful TUIs in Python, its addressable market is exceptionally narrow [[18](https://realpython.com/python-textual/), [19](https://www.linkedin.com/pulse/introduction-textual-building-modern-text-user-interfaces-nuno-bispo-idlpe)]. It is designed for the middle band of users: those who tolerate a terminal but find raw CLI output overwhelming. The primary target audience—the users who actively avoid terminals—are not meaningfully helped by a prettier terminal UI . The development cost and maintenance burden of building and supporting a Textual TUI may not be justified when a web wrapper can serve the same user group more effectively. If the web wrapper is chosen as the primary solution, the Textual TUI becomes redundant. It should only be considered as a potential optional extra (`hoard[tui]`) if there remains a demand from the niche audience of terminal-tolerant power users after the web interface is implemented .

## Deep Dive: Local Web Wrapper with FastAPI as the Primary Solution

The Local Web Wrapper strategy, centered on the FastAPI framework, is positioned as the primary solution for making HOARD accessible to non-terminal users. This approach creates a thin, local-only web application that serves as a graphical front-end to the core HOARD CLI, thereby addressing the interface adoption barrier while respecting the project's core design principles of minimalism and solo-maintainability. Its ranking is predicated on its broad reach, elegant solution to the cold-start problem, and lean architectural footprint.

From the target user's perspective, the experience begins with a simple double-click. After installation via the `uv` tool, the user finds a desktop icon named something like "HOARD Project Manager" . Upon clicking this icon, a terminal window briefly flashes open, starts the FastAPI server, and simultaneously opens their default web browser to `http://localhost:8765` [[124](https://stackoverflow.com/questions/60132045/fastapi-uvicorn-not-working-when-specifying-host)]. The user is now presented with a clean, modern web interface. They see a dashboard showing their projects, a prominent "New Project" button, and drag-and-drop zones for uploading their raw field records—context sheets, site photographs, and notes . With a few clicks, they can select a project template, choose a hardware profile (e.g., "My Laptop," "Office Desktop"), and initiate the pipeline. The interface then displays real-time progress bars for each of the six AI-assisted phases, logs of the processes, and visual previews of the generated reports before export . All interaction with the powerful backend is handled through this single, intuitive web interface, which abstracts away the entire complexity of the underlying CLI commands and file structures.

The cold-start problem is elegantly resolved through a two-part mechanism involving `uv` and a launcher script, as specified in the project goals . First, `uv` serves as the Python toolchain manager, acting as a drop-in replacement for pip and venv [[12](https://plotly.com/blog/uv-python-package-manager-quirks/)]. It simplifies the entire installation process into a single command that can be run by anyone comfortable with a terminal, collapsing the multi-step process of installing Python, setting up a virtual environment, and installing packages into a one-liner [[11](https://discuss.python.org/t/uv-another-rust-tool-written-to-replace-pip/46039), [54](https://dev.to/epam_india_python/meet-uv-the-next-gen-python-package-manager-built-for-speed-and-simplicity-3og7)]. Second, during the initial `hoard init` process, the application generates a platform-specific launcher script. On Windows, it creates a `.bat` file; on macOS, it creates a `.command` file [[51](https://stackoverflow.com/questions/62097824/running-a-python-script-by-double-clicking-desktop-icon-on-mac-os), [52](https://superuser.com/questions/966946/how-to-run-python-script-in-a-terminal-window-by-double-clicking-it)]. This script contains the exact command needed to start the local web server (`hoard serve`) and automatically opens the browser. The user interacts with this script via a double-click, completely avoiding manual command-line invocation for daily use. This approach avoids the high-maintenance pitfalls of background services or complex OS-level installers, aligning perfectly with the developer's constraints .

In terms of implementation within the HOARD codebase, several modules would need modification. The core `cli.py` would need to be extended to include a new `serve` subcommand. This command would be responsible for initializing the FastAPI application. New files would be created in a dedicated `web/` directory. This would contain the FastAPI application startup logic (`main.py`), route definitions for all UI interactions (e.g., `/projects`, `/upload`, `/phase/{id}/start`), and templating logic. The existing pipeline logic in `pipeline.py` and the data models in `models.py` would largely remain unchanged, as the web layer would interact with them indirectly via the CLI. The key change is creating a stable, programmatic interface for the web server to call the CLI commands and parse their structured outputs. The existing JSON-based file I/O patterns are ideal for this IPC model . The minimum viable product (MVP) would consist of a functional FastAPI server that can: 1) display a static project list, 2) accept file uploads via a form, 3) trigger a single pipeline phase via an API endpoint, and 4) display a basic progress indicator and log output streamed back from the CLI [[125](https://dev.to/jod35/building-a-chat-app-with-fastapi-and-javascript-using-javascript-3104)]. More advanced features like real-time progress updates via WebSockets and report previews could be added in subsequent iterations [[125](https://dev.to/jod35/building-a-chat-app-with-fastapi-and-javascript-using-javascript-3104)].

The dependency impact is a significant strength of this strategy. The FastAPI framework itself would be an optional dependency, installable via `pip install hoard[web]` . This ensures that the base `pip install hoard-erd` remains lean and focused on the core CLI functionality. The web UI would be decoupled from the main package, satisfying the constraint of a minimal core dependency footprint. For a solo developer, this modularity is crucial for long-term maintainability.

The estimated effort to a Minimum Viable Product (MVP) for a developer with existing FastAPI experience (as confirmed by the project's use of FastAPI in the Libby ecosystem component) is approximately 3-4 developer-weeks . This includes setting up the server, implementing the core CRUD operations for projects, integrating the file upload and pipeline execution logic, and building the basic frontend views. The use of HTMX is a strong candidate for the frontend approach, as it allows for building a rich, interactive UI with server-rendered HTML fragments, drastically reducing the need for complex client-side JavaScript and build toolchains [[43](https://blakecrosley.com/guides/fastapi-htmx)]. This keeps the development effort and ongoing maintenance low.

The main maintenance risk associated with the web wrapper is security. As a local web server, it introduces a small attack surface. The primary concerns are secure handling of the credential vault unlock and preventing any potential local file exposure. The vault unlock mechanism must be designed carefully; for instance, the password could be passed securely to the backend via a one-time token or a secure IPC channel, rather than being exposed in browser storage or logs [[38](https://stackoverflow.com/questions/44133536/is-it-safe-to-store-a-jwt-in-localstorage-with-reactjs), [128](https://stackoverflow.com/questions/14636290/secure-credential-storage-in-python)]. Since the application is strictly offline-first, the risk of external attacks is negligible, but internal vulnerabilities must be mitigated [[41](https://stackoverflow.com/questions/75573551/how-to-install-fastapi-on-an-offline-computer)]. Another risk is API evolution; as the core HOARD CLI evolves, the web wrapper's expectations of its output formats must be updated accordingly, requiring coordinated versioning between the two components [[147](https://www.sciencedirect.com/science/article/pii/S0164121224001559)]. However, given the controlled nature of the project and the developer's solo status, these risks are manageable and represent a known quantity compared to the unpredictable maintenance demands of a native desktop application or a divergent UI codebase.

## Deep Dive: Tauri Desktop Shell as the Secondary Solution

While the FastAPI web wrapper is the primary recommendation, the Tauri Desktop Shell strategy serves as a powerful secondary option, particularly for users who prioritize a native, single-installer application experience over a browser-based one. Tauri is a framework for building tiny, blazing fast binaries for all major operating systems using web technologies (HTML, CSS, JS) and a backend written in Rust [[62](https://dev.to/ottoaria/tauri-in-2026-build-cross-platform-desktop-apps-with-web-technologies-better-than-electron-11mo)]. Given the developer's existing Tauri 2 experience from the StratiGraph project, this option carries a lower initial learning curve and faster ramp-up time .

For the end-user, the Tauri solution feels indistinguishable from any other native application. The entire cold-start problem is solved by the installer. The user downloads a single `.dmg` file for macOS or `.exe` for Windows, drags the HOARD icon into the Applications folder, and launches it for the first time [[62](https://dev.to/ottoaria/tauri-in-2026-build-cross-platform-desktop-apps-with-web-technologies-better-than-electron-11mo)]. The experience is polished and consistent with modern software expectations. Inside the application, they see a window that hosts a web frontend—likely built with React, given the StratiGraph reference . This interface mirrors the functionality of the web wrapper: a project dashboard, file upload areas, controls for pipeline phases, and progress indicators. All communication with the outside world happens through this isolated window. When a user action requires the core HOARD functionality (e.g., starting a pipeline phase), the web frontend sends a message to the Rust backend via Tauri's Inter-Process Communication (IPC) mechanisms [[16](https://lobehub.com/skills/neversight-skills_feed-understanding-tauri-ipc)]. The Rust code then spawns a child process to execute the relevant `hoard` CLI command, streams the output back to the frontend in real-time, and presents it to the user. The terminal becomes entirely invisible infrastructure.

The implementation within the HOARD ecosystem would involve creating a new, separate repository for the Tauri application. This application would act as a shell around the Python package. A key architectural decision is how to handle the Python dependency. Bundling a full Python sidecar within the Tauri binary is possible but adds significant size and maintenance complexity [[13](https://stackoverflow.com/questions/75913627/how-to-use-tauri-app-and-python-script-as-a-back-end)]. A more pragmatic approach would be to assume Python is already installed on the user's system and use `uv` as the designated installation method, with the Tauri app simply calling the globally available `hoard` command. This makes the Tauri app much smaller but carries the risk of version mismatches. A middle ground could involve bundling `uv` as a sidecar binary within the Tauri application, which it then uses to ensure the correct version of the `hoard` package is available [[63](https://blog.csdn.net/zhgyuhui/article/details/161413146)]. The core of the Tauri app would be written in Rust for the backend logic and a web framework (like React) for the UI. The existing StratiGraph codebase could be reused for UI components and patterns, accelerating development .

Dependencies for this approach are split between the Tauri shell and the HOARD core. The Tauri shell has its own set of Rust dependencies. Crucially, it does not add any new dependencies to the base `hoard-erd` Python package itself. The `hoard-erd` package remains pure Python. The "dependency" here is the assumption that the Python environment is correctly set up, a problem solved by the `uv` installation method. The main new dependency is the Tauri toolchain (Rust, cargo), which the developer would need to manage.

The estimated effort to an MVP for a solo developer with Tauri experience is roughly 4-5 developer-weeks. This includes setting up the Tauri project, integrating the IPC calls to the `hoard` CLI, building the basic UI shell, and implementing the core workflow of project management and phase execution. The existing Tauri 2 experience is a significant advantage here, as it avoids the steep learning curve associated with Tauri 1 or other desktop frameworks like Electron [[94](https://electronjs.org/docs/latest/tutorial/security)]. The developer can leverage their knowledge of file handling, IPC, and web view integration from the StratiGraph project [[64](https://stackoverflow.com/questions/72322892/tauri-how-can-i-read-a-file-inside-the-project-folder), [65](https://quentinwach.com/blog/2024/11/26/files-in-tauri-v2.html)].

The primary maintenance risk for the Tauri solution is architectural divergence. The Tauri app and the `hoard` Python package are now two separate entities that must be kept in sync. Coordinating releases, managing breaking changes in the CLI that affect the Tauri shell, and maintaining two separate codebases are significant long-term burdens [[56](https://nesa.zju.edu.cn/download/auto_20260423_yifan-2026fse.pdf)]. This risk is higher than the web wrapper approach, where the frontend and backend are more tightly coupled in a single language ecosystem (Python). Additionally, Tauri, being a relatively young framework, may present challenges related to breaking changes or evolving best practices, although its stability has improved significantly since version 2 [[114](https://deprecated.readthedocs.io/en/latest/changelog.html)]. The complexity of managing a hybrid Rust and Python project also increases the cognitive load on the sole developer. Despite these risks, the superior user experience and complete abstraction of the terminal make it a compelling alternative, especially for users who are wary of applications that run servers on their machine.

## Strategic Synthesis and Final Recommendations

This research has systematically evaluated seven strategies for enhancing the accessibility of the HOARD archaeological reporting pipeline for non-terminal commercial archaeologists. The analysis reveals a clear, evidence-based path forward that prioritizes user reach, low maintenance burden, and adherence to the project's core architectural constraints. The investigation confirms that the primary challenge is not a high technical barrier, which is already mitigated by the existing Ultra-light hardware tier, but rather a dual barrier of installation friction and interface intimidation . The optimal solution requires a phased approach that begins with low-effort, high-impact procedural improvements and culminates in the development of a new graphical wrapper layer.

The foremost recommendation is to adopt the **Local Web Wrapper with FastAPI** as the primary accessibility strategy. This approach offers the most significant benefit to the largest segment of the target audience by providing a familiar, browser-based graphical interface. It elegantly resolves the cold-start problem through the combination of the `uv` installer and a double-clickable launcher script, fully complying with the need to eliminate terminal use for daily operations . Architecturally, it is the most sustainable option for a solo developer, as it keeps the core `hoard` package lean and modular, with the web components existing as an optional dependency (`hoard[web]`) . The developer's existing FastAPI experience from the Libby project further de-risks this path . The immediate next steps should focus on implementing foundational enhancements to improve the initial user experience. This includes developing a guided onboarding flow with an interactive `hoard wizard` and a `hoard doctor` command to check prerequisites, alongside the `uv`-based installation and launcher script mechanism . These initial actions will create a robust baseline of accessibility that can be layered upon.

As a secondary, complementary strategy, the **Tauri Desktop Shell** should be developed concurrently or in the near future. Leveraging the developer's existing Tauri 2 expertise from the StratiGraph project, this option provides a native, polished application experience that some users may prefer . It completely eliminates the cold-start problem by delivering a single installer, offering the smoothest possible user journey. While it introduces a slightly higher maintenance burden due to its hybrid architecture, the superior user experience justifies the additional effort. The Tauri shell would act as a sophisticated wrapper, communicating with the core HOARD CLI via IPC, thus keeping the business logic contained within the main Python package [[16](https://lobehub.com/skills/neversight-skills_feed-understanding-tauri-ipc)].

Other strategies play important, though more limited, roles. **Guided Onboarding** and **Configuration Presets** are essential, low-effort investments that should be implemented immediately. They directly address user confusion and reduce cognitive load without requiring new technology stacks, forming a critical foundation for all other UI efforts . The **Progressive Disclosure** approach, via a CLI wizard, is a viable fallback or intermediate step if a full web or desktop UI proves too complex, but it is less impactful as it still requires the user to operate within a terminal .

Strategies ranked lower should be approached with caution. **Trowel Integration**, while synergistic, is fundamentally incomplete as it can only function as a downstream editor for HOARD's outputs, failing to solve the critical first step of ingesting raw field data . Finally, the **Textual TUI Enhancement** is deemed the lowest priority. Its potential user base is too narrow to justify its development cost when a more effective web-based solution is available. It should only be considered as a niche, optional extra (`hoard[tui]`) for a specific segment of users who are comfortable with terminals but find the current Rich dashboard insufficiently detailed, and only after the primary web wrapper has been successfully implemented .

In conclusion, the path to unlocking HOARD's potential for the commercial archaeology sector is clear. By first solving the procedural and installation hurdles with `uv` and guided onboarding, and then building a modern, local-first web wrapper with FastAPI, the developer can create a powerful and accessible tool. This strategy respects the constraints of solo-maintenance, preserves the core strengths of the CLI-first architecture, and delivers a genuinely usable product to the intended audience.
