# Beyond Ollama: An Adaptive, Capability-Aware AI Abstraction for Hybrid Archaeological Reporting

## Architectural Blueprint for a Capability-Aware Provider Interface

The foundational requirement for evolving HOARD from a monolithic, Ollama-dependent pipeline into a flexible, hybrid reporting tool is the creation of a sophisticated yet practical provider abstraction layer. The primary objective of this layer is to enable interchangeability between local and cloud Large Language Models (LLMs) across the four distinct phases of archaeological report generation . The architectural decision hinges on a critical trade-off: should the abstraction prioritize maximum compatibility by exposing only the lowest common denominator of features, or should it pursue feature parity by actively exposing provider-specific capabilities through a unified API? The analysis of the project's goals and existing reference implementations decisively favors the latter approach. A successful design must move beyond simple compatibility to become a capability-aware router, allowing the system to intelligently select providers based on the specific requirements of each task . This philosophy is best exemplified by the production-tested multi-provider client within Simmer, which uses a single public method (`chat`) that internally routes calls to format-specific implementations based on an `ApiFormat` enum . This pattern provides the perfect blueprint for a unified interface that can be extended with a crucial `capabilities()` method, transforming it from a simple adapter into an intelligent orchestration layer.

The core of this architecture is a well-defined `ModelProvider` abstract base class (ABC). This interface serves as the contract for all concrete provider implementations, whether they connect to a local Ollama instance or a global cloud service. Its design must be centered on clarity, extensibility, and alignment with modern Python practices. Instead of a thin wrapper around disparate APIs, it will act as a full abstraction layer, translating generic requests into provider-specific payloads. The key innovation, however, is the introduction of a `capabilities()` method. This method will return a structured object detailing the provider's abilities, such as its support for structured output, vision inputs, and cost tracking . This declarative approach decouples the core application logic from feature-specific code, enabling the system to dynamically route tasks to the most suitable provider without hardcoding routing logic within the phase-specific scripts [[42](https://www.linkedin.com/pulse/using-llm-structured-outputs-routing-john-damask-bbdve), [47](https://dev.to/pockit_tools/llm-structured-output-in-2026-stop-parsing-json-with-regex-and-do-it-right-34pk)]. For example, when Phase 3 requires structured drafting, the router can query the `capabilities()` of available providers and automatically select one that supports it, such as OpenAI or Anthropic, over one that does not. This makes the system highly extensible, as adding support for a new provider merely involves implementing the `ModelProvider` interface and correctly populating its `capabilities()` response.

A direct translation of the Simmer reference pattern into Python would involve creating an enumeration for API formats, mirroring their `ApiFormat` enum . However, the proposed design elevates this by embedding the format information directly within the `capabilities()` response. This avoids the need for a separate match statement at the call site and integrates the routing metadata directly into the provider object itself. The `api_format` string returned by `capabilities()` can then be used for routing decisions, just as in the Rust implementation. This choice ensures that the system remains true to the proven pattern established in the ecosystem while adding the necessary intelligence to meet HOARD's unique requirements. The use of Pydantic models for defining structured outputs further solidifies this approach, as it provides a robust, standardized way to define schemas that can be validated on both the client and server sides, reducing the likelihood of malformed responses and improving reliability [[36](https://community.openai.com/t/structured-output-precision-accuracy-pydantic-vs-a-schema/1054410), [39](https://dev.to/klement_gunndu/stop-parsing-json-by-hand-structured-llm-outputs-with-pydantic-1pg0)].

The implementation of this interface will require careful mapping of the unified request parameters to the specific parameter names and payload structures of each target provider's API. For instance, the `temperature` parameter is universally understood, but the mechanism for requesting structured output varies dramatically. Ollama uses a top-level `format` parameter with a JSON schema, OpenAI employs a `response_format` object with a nested `json_schema`, Anthropic leverages `tool_use` with a tool definition, and Google utilizes `response_mime_type` and `generationConfig.responseSchema` [[8](https://openai.com/index/introducing-structured-outputs-in-the-api/), [32](https://arxiv.org/html/2602.18903v1), [40](https://lobehub.com/skills/terminalskills-skills-structured-output)]. The `ModelProvider` implementation for each service will be responsible for constructing the correct API payload from the standardized `InferenceRequest` object. Similarly, vision support requires handling different image input formats; Ollama expects a list of base64 strings in a `messages` array, while OpenAI requires content parts with `image_url` objects, and Anthropic uses a `source` block [[26](https://community.openai.com/t/issue-with-gpt-4-vision-preview-api-when-passing-base64-encoded-image/517532), [30](https://community.openai.com/t/error-sending-base64-image-data-in-api-request-to-gpt-4-vision-model/495864)]. By encapsulating these provider-specific transformations within each concrete class, the main application logic remains clean and agnostic to the underlying technology. This modular design also facilitates maintenance and adaptation, as changes to a provider's API will only necessitate modifications to a single class rather than cascading updates throughout the codebase. The following table summarizes the key architectural choices and their rationale, grounding the design in established patterns and addressing the project's specific needs.

| Architectural Component | Proposed Design Rationale |
| :--- | :--- |
| **Primary Interface** | `abc.ABC` named `ModelProvider` [[82](https://www.linkedin.com/posts/ryan-bergman-3665763_so-i-thought-i-would-post-where-exactly-i-activity-7429183163228233728-FuQo)]. Enforces a consistent contract for all providers. |

This architectural blueprint forms the bedrock of the entire multi-provider system. It establishes a clear separation of concerns, where the `ModelProvider` family of classes is solely responsible for the complexities of interacting with external APIs, while the rest of the HOARD application focuses on its domain logic: processing archaeological data and orchestrating the report generation pipeline. This design is not merely an incremental improvement; it represents a fundamental shift towards a more resilient, adaptable, and powerful system. It directly addresses the core problems identified in the research goal: inconsistent hardware, the need for higher-quality prose from cloud models, and the desire for a seamless transition from a purely local workflow to a hybrid one. By building on the proven patterns of sibling projects like Simmer and Kryptis, the design avoids reinventing the wheel and instead focuses on creating a robust, secure, and user-centric solution tailored specifically for the unique challenges of archaeological documentation [[81](https://pypi.org/project/anyllm/), [92](https://dev.to/alwil17/securepasswordcrypt-secure-aes-gcm-encryption-password-hashing-for-net-projects-kap)].

## Implementation of Unified Inference Interfaces and Data Models

To translate the architectural blueprint into a practical and implementable system, a set of concrete Python data models and type signatures is essential. These interfaces form the language through which HOARD's four phases communicate with the abstract `ModelProvider` layer. They must be comprehensive enough to capture all necessary parameters for every supported feature—structured output, vision inputs, and context management—while remaining flexible and easy to use. The design prioritizes type safety and clarity, leveraging Python's `typing` module and Pydantic for robust data validation. This section details the complete set of type signatures for the core interfaces, along with detailed implementation outlines for the initial set of required providers: Ollama, OpenAI, Anthropic, and Google Gemini. Each provider implementation will adhere to the `ModelProvider` ABC, demonstrating how the unified request is translated into provider-specific API calls.

The foundation of the communication protocol is the `InferenceRequest` data model. This TypedDict acts as a container for all parameters needed to make an LLM call, grouping related settings together logically. It includes the core `prompt` text, the `model_name` to be invoked, standard generation parameters like `temperature` and `max_tokens`, and the specialized features required by HOARD's phases. For structured output, the request incorporates a `schema` parameter typed as a `BaseModel`. This is a critical design choice, as it allows developers to define the desired JSON structure using familiar Pydantic syntax, which is then serialized and passed to the provider [[38](https://community.openai.com/t/pydantic-model-responses-api/1147202), [39](https://dev.to/klement_gunndu/stop-parsing-json-by-hand-structured-llm-outputs-with-pydantic-1pg0)]. For vision support in Phase 2, the request includes an optional `images` list containing base64-encoded strings of the images to be processed . Finally, a `use_chunking` boolean flag is included to control the fallback logic for managing large contexts, a technique currently employed in Phase 3 . The corresponding `InferenceResponse` TypedDict is designed to be equally comprehensive, returning the generated `content`, detailed `usage` statistics (tokens, etc.), and an optional `error` string for failed requests, providing a clear and consistent way to handle both success and failure cases [[89](https://docs.anthropic.com/en/api/messages-count-tokens)].

With the request/response contracts defined, we can outline the implementations for each provider. The `OllamaProvider` serves as the baseline, as it is already in use by HOARD. Its `capabilities()` method will declare `supports_structured_output=True` because it uses the `format` parameter for JSON schema, and `supports_vision=True` because it accepts a list of base64 images . Its `chat` method will take the `InferenceRequest`, map the parameters to the `ollama.chat` function signature, and pay special attention to the `options` dictionary, which maps to `temperature` and `keep_alive` . The `OpenAIProvider` will have similar capabilities, but its implementation will differ significantly. It will translate the Pydantic `schema` into the `response_format` object required by the OpenAI API, specifying `{"type": "json_schema", "json_schema": schema.model_json_schema()}` [[4](https://community.openai.com/t/dynamic-json-schema-output-format-for-assistant/975365), [8](https://openai.com/index/introducing-structured-outputs-in-the-api/)]. For vision, it will construct the message history with `role: "user"` and a `content` list containing both text strings and objects with `type: "image_url"` [[31](https://www.zhihu.com/en/article/666723793)]. The `AnthropicProvider` presents a slightly different paradigm. While it doesn't have native JSON mode, it supports structured output via `tool_use` [[40](https://lobehub.com/skills/terminalskills-skills-structured-output)]. Therefore, its `capabilities()` will reflect this, and its `chat` method will convert the Pydantic schema into a single tool definition for the Anthropic API. It may also need to parse responses that contain multiple text blocks and citations [[84](https://docs.anthropic.com/en/docs/build-with-claude/citations)]. The `GoogleProvider` will similarly leverage its `response_mime_type: "application/json"` and `generationConfig.responseSchema` fields to enforce structured output [[32](https://arxiv.org/html/2602.18903v1)]. For vision, it handles inline data parts [[33](https://dev.to/googleai/multimodal-rag-with-the-gemini-api-file-search-tool-a-developer-guide-5878)]. The following table provides a high-level summary of these mappings.

| Feature | Ollama | OpenAI | Anthropic | Google Gemini |
| :--- | :--- | :--- | :--- | :--- |
| **Structured Output** | `format: "json_object"` [[97](https://www.aidoczh.com/ollama/docs/reference/openai/)] | `response_format: {"type": "json_schema", ...}` [[8](https://openai.com/index/introducing-structured-outputs-in-the-api/)] | `tools` + `tool_choice` for `tool_use` [[40](https://lobehub.com/skills/terminalskills-skills-structured-output)] | `response_mime_type: "application/json"` [[32](https://arxiv.org/html/2602.18903v1)] |

Beyond the core functionality, a robust error handling strategy is paramount for a resilient hybrid system. The provider implementations must catch exceptions from their respective HTTP clients and translate them into a standardized set of error types that the HOARD application can understand. Network-related errors, such as connection timeouts or unreachable hosts, should trigger a retry mechanism with exponential backoff and jitter, as these are often transient [[52](https://stackoverflow.com/questions/75041580/openai-api-giving-error-429-too-many-requests)]. Errors indicating rate limiting, specifically HTTP `429 Too Many Requests`, must be handled gracefully. The system should pause and respect any `Retry-After` header provided by the server before attempting the request again [[52](https://stackoverflow.com/questions/75041580/openai-api-giving-error-429-too-many-requests)]. Authentication failures, signaled by a `401 Unauthorized` status code, require a specific handler that can notify the credential management system to check for valid keys [[51](https://community.openai.com/t/error-retrieving-completions-400-bad-request/34004)]. More nuanced errors, such as `400 Bad Request` due to an invalid JSON schema or an unsupported parameter, must be carefully diagnosed. These are typically developer or configuration errors and should result in a clear, descriptive error being raised to halt the current operation and prevent silent failures, rather than triggering a retry [[6](https://stackoverflow.com/questions/79039544/openai-api-error-invalid-parameter-response-format-of-type-json-schema-is), [90](https://community.openai.com/t/json-schema-required-in-api-request-despite-documentation-stating-otherwise/1029853)]. For context window overflow, the system should first attempt to apply chunking logic locally. If a cloud provider returns a similar error, it could indicate a smaller effective context window and might trigger a fallback to the chunking strategy, or it could signal a need for a larger-context model [[54](https://community.openai.com/t/openai-structured-output-allowed-types/1316209)]. This comprehensive error handling framework ensures that the abstraction layer not only translates requests but also manages the inherent instability of distributed systems, making the overall application more robust and reliable for the end-user.

## Intelligent Hybrid Routing and Fallback Mechanisms

A multi-provider abstraction is only as useful as its ability to intelligently select the right provider for the job. Simply offering a menu of options is insufficient; the system must embody a "smart router" that dynamically chooses a provider based on a combination of explicit user preferences, environmental conditions, task requirements, and real-time performance metrics. The design must accommodate three distinct operational modes as outlined in the research goal: Manual mode, where the user has full control; Auto mode, which relies on automated checks for hardware and network availability; and Quality mode, which optimizes for the best possible output for specific tasks . The optimal implementation is not to treat these as mutually exclusive modes, but rather to create a hierarchical, cascading decision-making process that leverages each in turn to arrive at the most appropriate provider.

The routing logic begins with **Manual Mode**, which takes absolute precedence. When a user specifies a provider for a given phase in the `config.yaml` file, the system must honor that choice without question . This provides ultimate control for power users who may have specific reasons for preferring one service over another. Once a manual configuration is confirmed, the system proceeds to instantiate the specified provider. The next step is to validate the provider's readiness. This involves checking for network connectivity (if a cloud provider is selected) and verifying the presence and accessibility of the required GPU resources. If a provider fails to initialize—for example, if the specified model is not available in Ollama or if the network is down—the system enters its **Fallback Mechanism**. This is a critical component for resilience, ensuring that a single point of failure does not bring the entire report generation process to a halt [[16](https://www.linkedin.com/posts/julio-pessan_ai-llm-machinelearning-activity-7437841416656928768-vwhF)]. The fallback chain is pre-configured based on the selected hardware tier and the capabilities required by the current phase. For instance, if Phase 3 is configured to use `claude-sonnet-4` but the service is temporarily unavailable, the router will immediately attempt to fall back to the next provider in the "Standard" tier's chain, such as a premium OpenAI model. If that also fails, it continues down the chain until a working provider is found or the chain is exhausted. This automatic failover should be accompanied by a warning logged to the user, informing them of the failure and the chosen fallback, maintaining transparency.

If no manual provider is configured for a phase, the system defaults to an **Auto Mode** orchestrated by the hardware tier system. On initial setup (`hoard init`), the system performs a series of auto-detections: it probes for available GPUs using libraries like `GPUtil` or `torch.cuda.is_available()`, checks for internet connectivity, and attempts to load the models recommended for each tier . Based on these findings, it suggests an appropriate tier (e.g., "Budget" for a machine with a 6GB VRAM GPU, "Ultra-light" for a CPU-only machine) and configures a default provider chain for each phase accordingly. For example, the "Ultra-light" tier's chain for Phase 3 synthesis would consist of cloud models like Gemini 2.5 Pro, as local models of sufficient quality are not feasible . The final step in the routing decision is **Quality Mode**, which refines the selection from the auto-mode chain. Before executing a request, the router filters the available providers in the chain based on the `capabilities()` of each one. If the current task is Phase 3 synthesis, which benefits from strong prose generation, the router will preferentially select a provider known for high-quality text, such as Claude or Gemini Pro, over a cheaper but potentially less capable alternative. If the task were Phase 1 document digitization, it might prioritize speed and accuracy over prose quality. This multi-layered approach ensures that the system is both resilient (via fallbacks) and optimized (via quality considerations), adapting intelligently to the user's environment and the demands of the task at hand. This intelligent routing mechanism is the heart of the hybrid system. It transforms HOARD from a static pipeline into a dynamic, adaptive engine. It directly addresses the core problem of supporting archaeologists in varied environments—from a field site with a powerful GPU to a university library with only a CPU—by ensuring that the software adapts to the available resources rather than forcing the user to adapt to the software. By combining explicit user control with automated heuristics for availability, hardware, and quality, the system achieves a balance that is both powerful and user-friendly, laying the groundwork for a truly seamless hybrid workflow.

## Secure Credential Management and Privacy-Preserving Architecture

Integrating cloud-based AI services introduces significant responsibilities concerning data privacy and security, which are central to HOARD's "local-first" ethos. The architecture must ensure that the addition of cloud capabilities does not compromise the trust placed in the system by its users. This requires two parallel, yet deeply interconnected, solutions: a robust credential management system that adheres to strict security standards, and a transparent, privacy-preserving architecture that gives users full control over their data. The design must guarantee that no sensitive information, such as proprietary archaeological findings or personal data, ever leaves the user's machine without their explicit consent and knowledge. This section details the implementation of a Kryptis-compatible credential vault and the architectural principles that underpin a trustworthy hybrid workflow.

The cornerstone of secure credential management is adherence to a proven cryptographic standard. The research goal mandates compatibility with Kryptis, a standalone credential vault that uses AES-256-GCM encryption with a PBKDF2 key derivation function . The recommended approach is to implement this same standard within HOARD's Python codebase, rather than creating a runtime dependency on the Kryptis project itself. This standalone implementation avoids coupling to potential Rust/Tauri dependencies, keeping HOARD's footprint pure Python and ensuring it runs reliably on headless Linux systems common in academic and field settings . The encryption process will follow the exact parameters specified by Kryptis: a master password will be used with PBKDF2-SHA256, 100,000 iterations, and a 16-byte random salt to derive a 32-byte AES-256 key [[12](https://stackoverflow.com/questions/67385361/is-this-password-function-with-sha256-and-pbkdf2hmac-salt-secure-to-encrypt-yo), [22](https://docs.delinea.com/online-help/secret-server/security-hardening/encryption-faq/index.htm), [92](https://dev.to/alwil17/securepasswordcrypt-secure-aes-gcm-encryption-password-hashing-for-net-projects-kap)]. For each individual API key, a new 12-byte random nonce will be generated to ensure semantic security with AES-GCM [[9](https://stackoverflow.com/questions/44009204/correct-nonce-iv-size-for-aes-gcm-mode), [63](https://stackoverflow.com/questions/67307689/decrypt-an-encrypted-message-with-aes-gcm-in-python)]. The encrypted key will be stored in a base64-encoded format containing the salt, nonce, and ciphertext, a scheme that is cross-compatible with Kryptis [[49](https://stackoverflow.com/questions/67963529/encrypt-in-js-front-end-and-decrypt-in-python-backend-using-aes-gcm)].

The credential storage itself will be a simple YAML file, `credentials.yaml.enc`, located in the global configuration directory (`~/.config/hoard/`). This file will store keys in a structured format that mirrors Kryptis's schema, including fields for `provider`, `value` (the encrypted key), `label`, and timestamps . To manage this vault, HOARD will introduce a dedicated CLI subcommand, such as `hoard cred`. This command will provide a secure interface for users to add, list, and remove API keys. When adding a key, the user will be prompted for the master password once, which will be held in memory during the session and never stored. The system will then prompt for the provider type (e.g., `openai`) and the API key itself, encrypt it, and append it to the vault file. This process ensures that API keys are never stored in plaintext in configuration files or version control . Furthermore, the `CredentialStore` service, which interacts with this vault, will be designed with zero-trust principles: it will never expose decrypted keys in logs, error messages, or other unencrypted contexts, preventing accidental leakage .

Complementing the secure credential system is the privacy-preserving architecture, which is built on the principle of radical transparency. The "local-first" promise means that all raw field data—including photographs, context sheets, and finds catalogues—must remain exclusively on the user's local filesystem at all times . The only data sent to a cloud provider is the minimal, processed information required for the inference task. For Phase 1 (document digitisation), this is the OCR prompt derived from the image. For Phase 2 (photo captioning), this is the prompt plus the base64-encoded image. Crucially, the system must include safeguards to prevent sensitive data from being inadvertently included in prompts. Before sending any request to a cloud provider, the system should scan the prompt for known markers of sensitive information (like GPS coordinates) and warn the user if such data is present. The concept of a granular "privacy tier" system, while mentioned, may be too complex for an initial release. A simpler, more robust approach is to provide a single, global `offline_mode` flag. When enabled, this flag instructs the router to completely ignore all cloud providers, forcing a purely local execution path regardless of configuration . This provides a definitive "kill switch" for privacy concerns.

Transparency is achieved through a detailed audit trail. Every inference request, whether local or cloud-based, must be logged with comprehensive metadata. This log, maintained per project, will serve as a permanent record of all AI activity. Each entry should include the phase, timestamp, provider name, model used, number of tokens sent/received, estimated cost, and a hash of the prompt's content (to identify the request without logging its full contents) . A typical log entry might look like: `PHASE_3 | 2026-06-08T14:30:00Z | 12,847 tokens | $0.19 | claude-sonnet-4 | Prompt hash: a1b2c3...`. This level of detail empowers users to verify that their data was handled according to their expectations and provides invaluable debugging information if something goes wrong. Finally, the architecture must support custom endpoints for institutional users who run their own private LLM deployments (e.g., vLLM or TGI) behind a corporate firewall . The `ModelProvider` abstraction naturally accommodates this by allowing a `base_url` to be configured for any OpenAI-compatible provider, enabling these users to seamlessly integrate their own infrastructure into the HOARD pipeline. Together, the secure credential vault and the transparent, user-controlled architecture form a defense-in-depth strategy that upholds HOARD's commitment to privacy and security in a hybrid world.

## Adaptive Offline Model Tiering and Hardware Abstraction

One of the most significant challenges in deploying AI applications on consumer hardware is the vast disparity in available resources. Archaeologists may use anything from a low-power laptop with no dedicated GPU to a high-end workstation with multiple professional graphics cards . A rigid system that assumes a specific hardware configuration is destined to fail for a large portion of its intended audience. The solution lies in an adaptive offline model tiering system that abstracts away the underlying hardware differences and presents the user with a simplified, yet powerful, configuration paradigm. This system will automatically detect available resources on first use and suggest an appropriate tier, while still allowing for manual override, thus balancing automation with user control .

The tier system is designed to map hardware capabilities to a predefined set of model configurations for each of HOARD's four phases. The research goal outlines a sensible progression of tiers: Ultra-light, Budget, Standard, and Performance . Each tier corresponds to a specific amount of available VRAM and dictates which models are used. The "Ultra-light" tier is for machines with no GPU, relying entirely on fast, efficient cloud models. The "Budget" tier targets machines with approximately 6GB of VRAM, matching the current Ollama setup. The "Standard" tier scales up to 8-12 GB of VRAM, allowing for more capable models like Qwen3-VL-8B. Finally, the "Performance" tier is for high-end machines (16-24 GB VRAM) that can run demanding models like PaliGemma 2 or Qwen3-8B . This tiered approach solves the "inconsistent hardware" problem by providing a curated, tested set of configurations rather than leaving the user to manually search for and install suitable GGUF or Ollama models [[18](https://www.cnblogs.com/farwish/articles/17643508.html), [43](https://dev.to/mitchell_cheng/my-learning-notes-choosing-the-right-ai-model-and-hardware-237)].

Implementation of this system begins with an auto-detection script run during the `hoard init` process. This script will perform several key checks. First, it will determine the presence and capacity of available GPUs. Using a library like `GPUtil` or checking `torch.cuda.is_available()`, it can identify if a CUDA-enabled GPU is present. If so, it can query the total and free VRAM to get a sense of the device's capacity. Second, it will perform a basic network connectivity test to determine if cloud services are accessible. Third, and most importantly, it will verify the availability of the models associated with each tier. This involves attempting to load the recommended model for the "Performance" tier, then the "Standard" tier, and so on, using the `llama-cpp-python` library for llama.cpp-based models or the Ollama client's health endpoint for Ollama-managed models . This proactive validation ensures that the suggested tier's models are actually installed and functional on the user's system. Based on these combined results, the system can make an informed recommendation. For example, if it detects a GPU with 8GB of free VRAM and finds the Qwen3-VL-8B model, it will suggest the "Standard" tier. It will then present this suggestion to the user in a clear, interactive prompt, asking for confirmation or providing an option to choose a different tier manually .

The interaction flow for the tier selection process is designed to be user-friendly and non-intrusive. Upon running `hoard init`, the system executes the detection script and generates a summary:

The user can simply press enter to accept the recommendation or type a different tier name if they have a specific preference. The final choice is then saved to the `config.yaml` file, alongside any explicitly configured providers, completing the initialization process. The tier system is designed to be extensible; users can later edit the configuration file to define their own custom tiers with specific models if the predefined ones do not suit their needs. This tiering system, integrated with the intelligent provider router, creates a cohesive and adaptive experience. It ensures that every user, regardless of their hardware, gets a configuration that is optimized for their specific situation, maximizing performance and minimizing the friction of setup and maintenance. It is a pragmatic solution to a complex problem, directly addressing the "inconsistent hardware" issue cited in the initial problem statement and paving the way for a more inclusive and capable archaeological reporting tool [[17](https://arxiv.org/html/2504.08791v2), [77](https://dl.acm.org/doi/10.1145/3754448)].

## Cost-Effective Hybrid Workflows and Migration Strategy

The integration of cloud-based AI services fundamentally alters the economic calculus of running HOARD. While local inference carries hidden costs in hardware acquisition and depreciation, cloud services offer a pay-as-you-go model that can be significantly cheaper for sporadic use cases . The research goal is to design a system that not only enables this hybrid workflow but also makes it transparent and cost-effective for the end-user. This requires a dual-pronged approach: first, designing a system that intelligently balances the quality-to-cost ratio of different providers, and second, providing a clear, incremental migration path from the current Ollama-only setup that minimizes disruption and risk.

The core of the cost-effectiveness argument comes from a detailed cost analysis. The preliminary research indicated that for a typical archaeological site report, the cost of using budget cloud models is substantially lower than the amortized cost of local hardware . This analysis must be updated and presented clearly to the user. The system will incorporate a cost tracker that estimates the cost of each inference request based on the provider's pricing, the number of tokens used, and the specific model. This data will be logged in the audit trail, providing users with real-time feedback on their spending . The intelligent routing mechanism is the key to realizing these savings. By default, the system will configure the router to prefer the most cost-effective provider that meets the minimum quality threshold for each phase. For example, Phase 1 and Phase 4 might default to the cheapest viable model (Gemini Flash-Lite), while Phase 3 might default to a mid-tier performer like Claude Sonnet 4. Users can always override these defaults in the configuration file to opt for higher-quality, higher-cost models when budget permits. This gives users control over the trade-off between cost and output quality.

To ensure a smooth transition from the current state, the migration strategy must be incremental and non-disruptive. The new multi-provider abstraction layer will be developed in parallel with the existing Ollama-only code. The initial implementation will focus on making the `ModelProvider` interface fully compatible with the current Ollama calls, ensuring that replacing `ollama.chat(...)` with `provider.chat(request)` is a straightforward substitution. The first release will introduce the new configuration file format (`config.yaml`) and the tier detection system, but it will default to using only the local Ollama provider unless explicitly configured otherwise. This allows users to upgrade to the new version of HOARD without any immediate changes to their workflow, benefiting from the improved architecture (e.g., better error handling, a cleaner codebase) while continuing to operate exactly as they did before. Subsequent releases will gradually introduce the cloud provider capabilities, starting with a small set of providers (OpenAI, Anthropic) and expanding over time. Each new feature, such as the credential manager or the audit trail, will be introduced incrementally, with clear documentation explaining how to enable and use them. This phased rollout minimizes the risk of breaking changes and allows users to adopt the new hybrid capabilities at their own pace, building confidence in the system as they go. This deliberate, user-centric migration plan is crucial for the successful adoption of the new, more powerful version of HOARD.
