# The Shared Schema and the Review Gate: An Architecture for an Interoperable, Offline-First Heritage Science Ecosystem

## A JSON Schema-Centric Data Model as the Foundation for Interoperability

The establishment of a shared data model is the foundational pillar upon which the entire interoperable heritage science ecosystem must be built. Without a common language for data, every integration becomes a brittle, ad-hoc mapping exercise prone to drift and decay, directly contradicting the user's goal of creating a sustainable, long-term solution maintainable by a solo developer . The analysis of available options reveals that a JSON Schema-centric approach, using it as the canonical source of truth while providing auto-generated native type packages as convenience wrappers, offers the most pragmatic and robust path forward. This design prioritizes zero external dependencies, leverages existing tooling, and provides a clear migration path, making it uniquely suited to the project's constraints of offline-first operation and multi-language support across Python, TypeScript, and Rust .

The primary rationale for selecting JSON Schema is its proven viability within the existing ecosystem. The successful bidirectional integration between HOARD and StratiGraph already relies on a shared `schemas/context-sheet-v1.json` and `schemas/context-relationships-v1.json`, validating the concept in practice . This working proof-of-concept eliminates the need for a high-risk, theoretical solution and provides a direct precedent for scaling the approach. Furthermore, JSON Schema aligns perfectly with the "zero dependencies" constraint for parsing; standard libraries in Python (`jsonschema`), TypeScript (`ajv`), and Rust (`json-schema-rs`) provide comprehensive validation capabilities out of the box [[30](https://pypi.org/project/jsonschema-rs/0.18.1/), [59](https://python-jsonschema.readthedocs.io/), [60](https://pypi.org/project/jsonschema/)]. Its text-based nature makes it ideal for the proposed file-based inter-process communication (IPC), where projects exchange data via JSON files in a shared workspace . In contrast, binary formats like Protocol Buffers, while offering superior performance, introduce a mandatory build step involving code generation and compilation . This adds significant friction to the development workflow, complicates dependency management, and runs counter to the goal of simplicity and ease of adoption for a solo maintainer .

The proposed architecture for implementing this model is designed for maximum flexibility and loose coupling. It consists of three key layers:
1.  **Source of Truth:** A dedicated `schemas/` directory will house versioned JSON Schema files. This directory acts as the single, authoritative definition for each data entity in the ecosystem, such as `context-sheet-v1.json`, `finds-v1.json`, `calibrated-date-v1.json`, and so on . This central repository ensures that any changes are explicit, versioned, and traceable.
2.  **Auto-Generated Native Packages:** To enhance developer experience without enforcing usage, two convenience packages will be generated from the canonical schemas. A Python package named `heritage-models` will provide Pydantic models, offering compile-time type checking, IDE autocompletion, and seamless serialization/deserialization to and from JSON . Similarly, a TypeScript package named `@heritage/types` will provide TypeScript interfaces and AJV validation functions, serving the same purpose for frontend-heavy projects like StratiGraph and Libby . These packages act as "sugar," improving productivity for developers who choose to use them.
3.  **Optional Build Step:** A simple, optional command, such as `make generate-models`, will be used to regenerate both the Python and TypeScript packages whenever the canonical schemas in the `schemas/` directory change . Projects that do not wish to use these native packages can continue to parse the JSON files directly, maintaining a completely decoupled relationship with the model packages .

The minimal set of shared types must be comprehensive enough to cover the core data concepts required by all listed projects. This includes `StratigraphicUnit` (or `ContextSheet`), `Find` (with sub-types for material, type, weight, and count), `Sample` (encompassing environmental, C14, and isotopic data), `Photo`/`Drawing`, `Site`/`Project` metadata, `StratigraphicRelationship`, and `Chronology` (for calibrated dates and SPD data) . For cross-referencing, string-based UUIDs are the recommended method. This approach is widely adopted in loosely coupled systems as it avoids tight coupling to specific database identifiers or memory addresses, ensuring that references remain valid even as data moves between different applications and storage backends . For instance, a `Find` would contain a `context_id` field with the UUID of the `StratigraphicUnit` it belongs to.

Schema evolution is a critical consideration for a long-lived system. The strategy must prioritize forward compatibility to prevent breaking older versions of software when newer data is introduced. All parsers within the ecosystem should be configured to ignore unknown fields, allowing a project compiled against version 1 of a schema to safely process data produced by a newer version that may have added optional fields . Versioning is handled explicitly through the filename itself (e.g., `-v1.json`, `-v2.json`), providing a clear contract for consumers of the data . When a breaking change is necessary, a new version of the schema is created, and a migration script can be provided to convert data from the old structure to the new one. This disciplined approach, combined with a governance process for proposing and reviewing changes to the `schemas/` directory, will mitigate the risk of schema drift over time.

Finally, the choice of IPC mechanism is dictated by the offline-first principle. File-based IPC is the only viable option, as it requires no network connections, servers, or persistent background processes . The orchestration layer will manage a shared workspace for a given archaeological project. After an automated phase completes, a project writes its output to a standardized directory within this workspace (e.g., `project_x/hoard_phase1_output/samples.json`). The subsequent project in the pipeline then reads its input from this location. This simple, decoupled model is resilient, easy to debug, and fully compatible with the ecosystem's operational requirements. The following table summarizes the trade-offs considered for the shared data model.

| Option | Pros | Cons | Recommendation |
| :--- | :--- | :--- | :--- |
| **JSON Schema + Native Wrappers** | Zero external dependencies for parsing; proven in existing integrations; language-agnostic; supports both file-based IPC and in-process use; auto-generation simplifies maintenance. | Slightly more verbose than binary formats; requires careful management of schema evolution. | **Recommended**. Best balance of simplicity, power, and adherence to project constraints. |
| **Protocol Buffers / Cap'n Proto** | High performance; strong typing and code generation; compact binary format. | Requires a build step and code generation; introduces external toolchain dependencies; less human-readable for debugging. | Not recommended for initial implementation due to increased complexity for a solo developer. |
| **Language-Specific Packages** | Provides the best developer experience for a single language. | Creates tight coupling and fragmentation; does not solve the cross-project problem. | Not recommended as the sole solution; the native packages are a secondary benefit, not the primary mechanism. |

This JSON Schema-based model provides a stable, extensible foundation. It allows the ecosystem to grow organically, with new projects adopting the shared schemas from day one and existing projects migrating incrementally. It directly addresses the core problems of schema drift and duplicated effort, paving the way for the higher-value goals of pipeline orchestration and unified services.

## Checkpoint-Based Orchestration for Human-in-the-Loop Archaeological Workflows

With a shared data model established, the next critical component is a pipeline orchestration layer capable of executing complex, end-to-end archaeological workflows. The analysis indicates that a purely automated, batch-mode execution is insufficient for this domain. Archaeological reporting is not merely a data transformation task but a deeply interpretive synthesis that requires expert human judgment at several key stages [[7](https://www.researchgate.net/publication/404401587_Clarifying_stratigraphic_terminology_in_Palaeolithic_archaeology_using_Natural_Language_Processing)]. Therefore, the optimal architecture is a checkpoint-based system that incorporates mandatory human review gates, balancing automation with interactive validation. This hybrid approach acknowledges the role of technology as an enabler of expert work rather than a replacement for it, a principle supported by research on human-in-the-loop (HITL) systems in complex domains [[26](https://arxiv.org/html/2604.16520v1), [27](https://www.linkedin.com/posts/bhavishyabharadwaj_ai-rag-hellopm-activity-7438821244474466304-m3AZ), [54](https://www.linkedin.com/posts/jatinsoni08_springframework-springcore-ioc-activity-7455914142319890432-26TK)].

The proposed orchestration pattern is not a single, monolithic command but a flexible framework that can accommodate both simple, internal workflows and complex, cross-project pipelines. For the majority of users who primarily interact with HOARD, the existing `hoard run` command can be extended with new flags. A `--auto` flag would allow for fully automated execution, useful for testing or CI environments where manual intervention is impossible . Conversely, the default behavior would pause the pipeline at designated review gates. A `--resume` or `--continue` flag would allow a user to proceed after completing the manual review steps . This preserves the familiar interface for the core user base while adding powerful new capabilities.

For the full ecosystem workflow, which spans multiple disparate applications, a declarative approach is superior. A YAML file, `pipeline.yaml`, would define the entire sequence of operations in a transparent and auditable manner . This file would specify a series of steps, which can be either automated phases within a single project or calls to other projects, interspersed with explicit review gates. An example workflow might look like this:

```yaml
steps:
  - project: hoard
    phases: [0, 1, 2] # Fully automated digitization and triage
  - gate: review
    message: "Review Harris Matrix in StratiGraph before proceeding with dating."
    action: stratigraph import --file hoard_phase2_output/matrix.json
  - project: libby
    action: calibrate-samples
    input: hoard_phase2_output/samples.json
  - project: hoard
    phases: [3, 4, 5] # Automated synthesis, compliance check, and final report generation
```

This declarative format makes the complex workflow explicit, allowing the user to understand, modify, and rerun it easily. It transforms the pipeline from a black box into a documented scientific procedure. The orchestrator would execute this plan sequentially, managing the flow of control and data between steps.

Data transport between these distinct projects remains strictly file-based, adhering to the offline-first and dependency-free principles. After `hoard` completes its first set of phases, it writes its structured output to a well-defined location in the project's workspace, such as `output/hoard-phase-2/`. The orchestrator then signals the completion of this step. If the next step is a human review gate, it prints an actionable message to the terminal, prompting the user to perform a specific action, such as "Run `stratigraph open` to visualize the matrix." Once the user completes the task, they can invoke the orchestrator with `--resume` to proceed. If the next step is an automated project like `libby`, the orchestrator invokes the `libby` executable, passing it the path to the input data (`output/hoard-phase-2/samples.json`) as a command-line argument. This decouples the projects entirely; `libby` does not need to know about `hoard` or the orchestrator, it simply reads from a known file path.

State management and resumability are crucial for a workflow that may span days or weeks. The orchestrator itself will maintain a global state file, tracking which steps have been completed. However, individual projects must also be inherently stateful. HOARD already implements a `PipelineState` JSON file that tracks progress within its own phases . The orchestrator will respect this internal state. If a pipeline is paused and restarted, the orchestrator checks its own state to determine the next step, and that step's corresponding project will load its own state file to resume from the correct point. This dual-layer state management ensures robustness against interruptions. The orchestrator could also implement a `checkpoint` action to manually save the state of the entire pipeline at any point.

The failure model must be carefully defined. If a required project is not installed or is inaccessible, the orchestrator has several options. For a production run, it should abort immediately with a clear error message, as the workflow cannot proceed correctly . For a `--auto` or testing run, it could attempt to skip the failed step, perhaps by inserting placeholder data (e.g., `[MISSING: Calibrated date — Libby not available]`) into the downstream report draft . This graceful degradation prevents a minor issue from derailing the entire process in a controlled environment. The orchestrator should also be able to list all available projects and their capabilities, allowing it to make intelligent decisions about which steps to execute.

The following diagram illustrates the data flow and control logic of the checkpoint-based orchestration system.

```mermaid

    subgraph "Orchestrator"
        direction TB
        A[Start Pipeline] --> B{Step Type?}
        B -->|Automated Phase| C[Execute Project]
        B -->|External Tool Call| D[Invoke External Executable]

        C --> F[Write Output to Workspace]
```

This architecture successfully navigates the tension between automation and human expertise. It automates the tedious, repetitive parts of the workflow, freeing up the archaeologist to focus on the high-value tasks of interpretation, validation, and synthesis. By making the review gates explicit and integral to the process, it creates a system that is not just efficient but also scientifically rigorous, producing outputs that are vetted by a human expert at every critical juncture.

## Unified Jurisdiction Templates and Vocabulary Services for Standardization

Beyond data models and workflows, achieving true interoperability in a diverse ecosystem requires standardizing the specialized knowledge and configurations that differentiate tools. Two key areas for standardization are jurisdiction templates, which dictate the structure and content of reports, and vocabulary, which ensures consistent terminology. The proposed solutions for both are built on the principles of a shared core with explicit extensions, allowing for both portability and specialization.

For jurisdiction templates, the current situation—14 templates in HOARD and 19 in Trowel with partial overlap—is untenable . A rigid, mutually exclusive schema would force compromises, leaving either tool with unused configuration fields or stripping away essential functionality like Trowel's report branding features . The superior solution is to design a unified template format based on a mandatory shared core, augmented by tool-specific extension blocks. This approach preserves the power of each application while ensuring that the fundamental report structure is portable across the ecosystem .

The proposed template format is a YAML file residing in a common directory, such as `~/.config/heritage/templates/` . Each template begins with a mandatory core section that both HOARD and Trowel must understand. This core defines the jurisdiction's identity, its required sections, and basic style rules. Following the core, there are two named extension blocks: `hoard:` and `trowel:` . Any configuration specific to HOARD, such as LLM prompt templates or extraction schemas, goes into the `hoard` block. Trowel-specific settings, like cover page templates or report numbering formats, belong in the `trowel` block. The key design rule is that tools must fail gracefully when encountering unknown keys at the top level, issuing a warning but continuing to process the understood core and any compatible extension blocks . Each extension block also contains its own `version` field, allowing for future migrations without affecting the stability of the core schema . The ordered list of `mandatory_sections` defines the canonical document structure, which is particularly important for Trowel during report generation . This design elegantly solves the divergence problem, enabling a single template file to serve both tools effectively.

The second pillar of standardization is a unified vocabulary service. Cache & Carry already provides a powerful offline backend, offering sub-millisecond lookup performance from a local SQLite database containing Getty AAT, ULAN, and TGN vocabularies . While Trowel already uses this for autocomplete, the service's potential extends much further. The proposal is to expose this capability through a simple, lightweight interface that can be consumed by any project in the ecosystem. The interface should be minimal, likely a single `search(term, vocabulary, limit)` function call that returns a structured list of matching terms with their labels and hierarchical relationships .

This service is more than just a lookup tool; it is a powerful normalization engine. Currently, HOARD's AI-driven context sheet digitization extracts materials and periods as free-text strings, leading to inconsistencies like "flint," "Flint," and "flint stone" for the same concept . By integrating the vocabulary service, HOARD can pass these raw extractions to the service for normalization. The service would map them to a single, controlled URI from the Getty vocab (e.g., "Flint/Chert" -> AAT: 300011754), dramatically improving data quality and enabling more sophisticated queries and analyses downstream [[17](https://dataliberate.com/2014/02/21/getty-release-aat-vocabulary-as-linked-open-data/), [37](https://www.researchgate.net/publication/291585812_Development_of_the_Getty_Vocabularies_AAT_TGN_ULAN_and_CONA)]. This directly addresses the problem of inconsistent terminology extraction.

The vocabulary service should also handle the complexities of archaeological thesauri. Periods, for example, are often hierarchical (e.g., Bronze Age > Hallstatt) and geographically dependent (e.g., Hallstatt culture in Germany vs. Britain) [[32](https://www.mdpi.com/2673-8945/5/4/105)]. The Getty vocabularies are designed to represent these relationships through broader/narrower links and other semantic constructs, which the service's query interface can leverage to provide context-aware suggestions [[19](https://agrovoc.fao.org/sparql)]. Furthermore, the service could eventually be expanded to include unit normalisation, mapping terms like "centimeter" and "cm" to a standard SI unit, though this is a more advanced feature .

The service itself can be implemented as a simple library/SDK that projects import and call directly, avoiding the overhead of a network server or complex IPC protocol. Given that Cache & Carry is written in Rust, the service could be a shared library (`.rlib`) that is linked into the various applications, or a small Python/TypeScript wrapper around the core Rust logic. This keeps dependencies low and performance high, consistent with the offline-first ethos. The following table outlines the key features of the unified vocabulary service.

| Feature | Description | Benefit |
| :--- | :--- | :--- |
| **Offline First** | Uses Cache & Carry's local SQLite database. No network connection required [[40](https://dev.to/daliskafroyan/builing-an-offline-first-app-with-build-from-scratch-sync-engine-4a5e)]. | Ensures usability in remote field conditions and protects data privacy. |
| **High Performance** | Sub-millisecond lookup times enable real-time UI interactions like autocomplete [[8](https://www.tiktok.com/@gtaneews/video/7320510531490549024?lang=zh-Hant)]. | Enhances user experience in tools like Trowel and potential future editors. |
| **Normalization Backend** | Maps free-text inputs to controlled Getty URIs, cleaning and standardizing data from tools like HOARD [[38](https://www.semanticscholar.org/paper/Development-of-the-Getty-Vocabularies%3A-AAT%2C-TGN%2C-Harpring/4067581ed2569b62574a44aacfb14873fe9fa7c7)]. | Improves data quality, consistency, and enables advanced semantic queries. |
| **Hierarchical Support** | Leverages Getty's RDF/SPARQL structure to navigate broader/narrower term relationships [[52](https://www.w3.org/wiki/SparqlImplementations)]. | Provides context for archaeological periods and materials. |
| **Simple API** | Minimalist interface (e.g., `search()`) reduces integration friction for developers [[44](https://stackoverflow.com/questions/77588061/typescript-to-json-schema-or-vice-versa-with-code-completion-navigation-in-int)]. | Lowers the barrier to entry for other projects in the ecosystem to adopt the service. |

By creating a unified vocabulary service, the ecosystem gains a centralized source of truth for its terminology. This not only improves the quality of individual tools but also creates a richer, more interconnected dataset that can be analyzed holistically. When every find, sample, and context sheet is tagged with standardized, machine-readable concepts from the Getty vocabularies, the collective research potential of the archived data increases exponentially. This combination of a flexible template system and a powerful normalization service completes the foundational layer of the ecosystem, enabling true semantic interoperability.

## Cross-Project Command-Line Interface and Implementation Roadmap

The final components of the proposed architecture are a unified cross-project Command-Line Interface (CLI) and a concrete, phased implementation roadmap. The CLI serves as the primary user-facing entry point to the orchestrated ecosystem, abstracting the underlying complexity of calling multiple disparate executables. The roadmap provides a practical, incremental path for a solo developer to build and deploy these components, starting with the highest-priority items.

For the CLI, the decision is between extending existing commands or creating a new top-level command. Creating a new `heritage` command is the recommended approach . It provides a clean namespace, signaling to the user that they are interacting with a coordinated system rather than a collection of independent tools. This unified command can intelligently route subcommands to the appropriate underlying project. For example, `heritage run --project X` would delegate to `hoard run`, while `heritage calibrate --project Y` would launch the calibration workflow, which internally calls `libby` . This abstraction layer hides implementation details and provides a consistent user experience.

The Python packaging strategy for this CLI is also critical. A separate `heritage-cli` package is preferable to a meta-package that re-exports entry points . A dedicated package allows for independent development, testing, and release cycles. It can declare dependencies on the core ecosystem projects (like `hoard`, `libby`, etc.), ensuring that the CLI is always compatible with the versions of those projects it is designed to control. This modular approach is easier to maintain and scale than a monolithic package that tightly couples the CLI to the lifecycle of every other project.

Configuration management should be centralized to simplify the user experience. A single configuration file at `~/.config/heritage/config.yaml` can store global settings, paths to external resources, and potentially default arguments for various commands . This avoids scattering configuration across multiple dotfiles in a user's home directory. For discoverability, the `heritage` command should include a `tools list` subcommand. This would scan the system for installed heritage science projects, report their status (e.g., installed, but requires a license), and describe their available commands, providing a dynamic and self-updating overview of the available ecosystem.

The following table maps the proposed unified CLI commands to the existing ones, demonstrating the abstraction and enhancement provided.

| Existing Command | Proposed Unified Command | Rationale |
| :--- | :--- | :--- |
| `hoard run --project X --phase 1` | `heritage run --project X --phase 1` | Unchanged; remains the primary workflow driver for HOARD. |
| `hoard calibrate` | `heritage calibrate --project X` | Abstracts the cross-project call to Libby behind a simple, memorable name. |
| `hoard lithics` | `heritage lithics --project X --input ./scans/` | Extends the HOARD namespace to new, planned functionality. |
| `trowel open --project X` | `heritage review --project X` | Uses a more descriptive verb ("review") that aligns with the checkpoint-based workflow. |
| `stratigraph import` | `heritage matrix --project X` | Groups matrix-related actions under a logical subcommand. |
| `hoard export --format docx,pdf` | `heritage publish --project X --format docx,pdf` | Uses a more semantically precise verb ("publish") for the final output stage. |

With the architecture defined, an incremental implementation roadmap is essential for feasibility. The following phased plan prioritizes deliverables based on their strategic importance and implementation complexity, allowing for tangible progress to be made in manageable chunks.

**Phase 1: Establish the Core Data Model (Highest Priority)**
1.  **Action:** Create the `schemas/` directory and commit the existing shared schemas from HOARD and StratiGraph (`context-sheet-v1.json`, `context-relationships-v1.json`) .
2.  **Action:** Write scripts to generate the initial `heritage-models` Python package and `@heritage/types` TypeScript package from the canonical schemas using tools like `datamodel-code-generator` and `json-schema-to-ts` .
3.  **Outcome:** A new, public `heritage-models` package is published to PyPI. The `@heritage/types` package is ready to be published. Other projects can now begin importing and using these shared types.

**Phase 2: Implement the Unified Vocabulary Service**
1.  **Action:** Create a new Rust library crate that provides a simple, safe API for querying the Cache & Carry SQLite database. This crate will be used by Cache & Carry.
2.  **Action:** Write a thin Python wrapper around the Rust library to make it accessible to Python-based projects like HOARD and Trowel.
3.  **Action:** Update Trowel to use this new library for its vocabulary lookups instead of directly accessing the SQLite DB.
4.  **Action:** Modify HOARD Phase 1 to use the vocabulary service for normalizing extracted materials and periods before storing them.

**Phase 3: Develop the Cross-Project CLI and Template System**
1.  **Action:** Create the `heritage-cli` Python package with a basic command structure.
2.  **Action:** Implement the `heritage run` command, initially just as a passthrough to `hoard run`.
3.  **Action:** Define the YAML schema for the unified jurisdiction templates and implement the loading/validation logic in both HOARD and Trowel.
4.  **Action:** Update both HOARD and Trowel to read and write the new unified template format, supporting the `hoard:` and `trowel:` extension blocks.

**Phase 4: Build the Full Pipeline Orchestration Layer**
1.  **Action:** Refactor the `heritage run` command to understand the `pipeline.yaml` format.
2.  **Action:** Implement the core logic for pausing at gates, printing messages, waiting for user input, and resuming execution.
3.  **Action:** Add support for invoking external executables (like `libby`) and passing them file paths as arguments.
4.  **Action:** Integrate the `heritage` CLI with the newly developed vocabulary service and unified templates.

This phased approach ensures that each new piece of the architecture builds upon a solid foundation. It delivers value early (Phase 1) and systematically tackles the remaining challenges, all within the realistic scope of a solo developer's capacity. The result will be a cohesive, interoperable ecosystem that significantly enhances the efficiency and quality of archaeological research.

## Strategic Synthesis and Risk Mitigation for Solo-Developer Maintenance

This research report has detailed a comprehensive architectural blueprint for unifying a decentralized, offline-first heritage science ecosystem. The proposed solution is centered on four pillars: a JSON Schema-centric data model, a checkpoint-based orchestration system with human review gates, a unified vocabulary service, and a cross-project CLI. This architecture is not merely a technical specification but a strategic framework designed to address the core problems of schema drift, duplicated effort, and fragmented workflows, all while respecting the stringent constraints of being maintained by a single developer. The success of this initiative hinges on the disciplined execution of an incremental implementation plan and a proactive approach to risk mitigation.

The cornerstone of the entire architecture is the shared data model built upon JSON Schema . This choice was driven by its proven utility in existing integrations, its zero-dependency nature, and its suitability for file-based IPC . By establishing a canonical source of truth in a `schemas/` directory and providing auto-generated native type packages as convenient wrappers, the design achieves a rare balance of strict contractual agreement and developer-friendly ergonomics. This foundation is what makes all subsequent interoperability possible. Without it, any attempt at integration would be a fragile patchwork of ad-hoc converters, doomed to break with each new update.

The second pillar, the checkpoint-based orchestration system, represents a profound understanding of the domain. It correctly identifies that archaeological synthesis is a human-in-the-loop process, not a fully automated one [[26](https://arxiv.org/html/2604.16520v1), [27](https://www.linkedin.com/posts/bhavishyabharadwaj_ai-rag-hellopm-activity-7438821244474466304-m3AZ)]. By designing a system that automates the tedious and then pauses for expert validation at critical junctures, the architecture elevates the tools from mere processors to collaborative partners. This pattern ensures scientific rigor by embedding human judgment directly into the workflow, a feature that is far more valuable than pure speed. The ability to define these workflows declaratively in a `pipeline.yaml` file adds a layer of transparency and auditability that is essential for scholarly work.

The third and fourth pillars—the unified vocabulary service and the cross-project CLI—serve to complete the ecosystem. The vocabulary service, built upon the existing Cache & Carry database, provides a powerful normalization engine that cleans and standardizes data at the point of creation, enhancing the quality of the entire dataset [[17](https://dataliberate.com/2014/02/21/getty-release-aat-vocabulary-as-linked-open-data/)]. The unified CLI provides a consistent, user-friendly facade over the complex underlying machinery, hiding implementation details and presenting a coherent system to the end-user . Together, they form a cohesive user experience that belies the distributed and heterogeneous nature of the software stack.

However, this ambitious vision carries inherent risks, particularly concerning maintenance burden. The primary risk is the cognitive load of managing a growing number of interconnected components. To mitigate this, the architecture is designed for loose coupling and modularity. File-based IPC, graceful handling of unknown data, and clearly defined schema boundaries all reduce the blast radius of changes. A change in the `CalibratedDate` schema, for instance, is unlikely to impact a project that only deals with `ContextSheet` data. Furthermore, the reliance on static analysis and build-time code generation (for the native type packages) shifts complexity away from runtime, which is generally easier to manage and debug.

Another risk is the potential for the architecture to become overly prescriptive, stifling innovation. The design actively counters this by favoring extension over inheritance. The use of explicit extension blocks in jurisdiction templates (`hoard:`, `trowel:`) is a prime example of this philosophy in action . It empowers each tool to evolve its own unique features without needing to negotiate changes to a central, monolithic schema. This decentralized approach to evolution is crucial for the long-term health of an open-source ecosystem.

In conclusion, the proposed architecture provides a clear, practical, and highly effective path toward achieving the user's goal. It is grounded in the realities of the existing ecosystem, respects the non-negotiable constraints of offline-first operation and solo-developer maintenance, and is strategically aligned with the interpretive nature of archaeological science. By prioritizing the data model, embracing human-in-the-loop validation, and building incrementally, the developer can construct a powerful and enduring platform. This platform will not only streamline their own workflow but also create a valuable, interoperable toolkit for the wider heritage science community, fostering collaboration and enabling cumulative research on a scale previously unattainable with siloed applications [[7](https://www.researchgate.net/publication/404401587_Clarifying_stratigraphic_terminology_in_Palaeolithic_archaeology_using_Natural_Language_Processing)].
